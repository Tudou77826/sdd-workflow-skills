# SDD 模块 ASIS 分析

## 目标

为目标模块中与本次需求相关的区域产出有代码证据支撑的 ASIS 分析，并写入当前 SDD 工作流指定的模块详细设计类成果物，使后续 TOBE 设计、门禁和 AICoding 能够复用同一组 ASIS 事实与证据。

这个 skill 不是用来编写泛化的模块说明书，而是用于支撑模块 TOBE 详细设计和后续编码实现。ASIS 只做现状事实发现，不做最终方案设计。

## 成果物协作规则

遵循当前 SDD 工作流或编排 skill 指定的成果物归属规则。对于本 skill，默认协作目标是模块详细设计类成果物；如果工作流没有另行指定，则使用 `xxx模块详细设计说明书`。

- 如果用户或工作流已提供目标成果物路径，更新该成果物中的 ASIS 相关章节。
- 如果目标成果物尚不存在，创建或输出该成果物的 ASIS 章节草案，并预留后续 TOBE、测试设计、风险与门禁章节的位置。
- 不要在没有依据的情况下另起一份孤立的 ASIS 文档；只有当工作流、用户或成果物拆分规则要求时，才拆分输出。
- ASIS 章节中的证据编号、需求/AR 映射编号、待确认问题编号，后续 TOBE 和门禁必须能够复用。
- 更新已有成果物时，保留非 ASIS 章节内容；只补充或修正与 ASIS 有关的章节、证据索引和待确认问题。

## 范围原则

默认分析范围是 `需求相关的模块内 ASIS`。

- 从用户提供的需求、AR、功能点、影响性分析、模块间交互设计或计划变更出发。
- 识别目标模块中受本次需求影响的部分。
- 只分析会影响后续 TOBE 设计、实施任务、风险控制或测试设计的模块区域。
- 为解释行为或隐藏约束，可以向相邻文件、入口、依赖、配置、测试或跨模块交互扩展。
- 只有在用户明确要求完整模块 ASIS，或影响范围无法被可靠收敛时，才扩展到完整模块级 ASIS。

如果分析范围发生扩展，必须说明扩展原因。

## 输入

可以接受以下任意组合：

- 当前 SDD 工作流指定的模块详细设计类成果物路径或当前草稿内容；未指定时默认为 `xxx模块详细设计说明书`。
- 目标代码仓路径。
- 需求、功能点、用户故事、缺陷、AR 设计项或计划变更。
- 目标模块名称、目录、包路径、服务、类、API、Job、Consumer 或功能区域。
- 上游 SDD 产物，例如需求分析、功能设计、AR 设计、模块影响性分析、模块间交互设计。
- 用户关注的表、接口、消息主题、配置项、迁移、风险或行为。

如果代码仓可访问，必须直接检查代码。若关键信息缺失但能从仓库中合理推断，可以继续分析并显式标记假设。只有在目标需求/变更或模块边界无法合理识别时，才向用户追问。

## 模块边界规则

把 `softWare.md` 作为第一优先级的模块边界来源。

1. 在识别模块前，先搜索仓库中的 `softWare.md`。
2. 阅读相关 `softWare.md` 条目，识别模块名称、目录、包路径、归属和声明的关系。
3. 用 `softWare.md` 确认模块边界，再用本次需求或 AR 收窄模块内 ASIS 分析范围。
4. 如果用户指定的模块与 `softWare.md` 冲突，必须记录双方信息并解释冲突。
5. 如果 `softWare.md` 缺失、不完整、疑似过期或含糊，则基于 import、包结构、依赖声明、测试、配置、路由、Job、Consumer、构建文件等推断边界，并标记为 `推断`。
6. 模块边界只表达归属关系，将文件和组件分类为 `确定属于模块`、`疑似属于模块` 或 `外部依赖`。
7. 本次分析切片单独表达相关性，将文件、组件或行为分类为 `需求相关`、`疑似相关` 或 `本次不分析`。

边界来源必须显式标记为：

- `declared`：由 `softWare.md` 或团队正式模块定义直接声明。
- `inferred`：未找到正式声明，基于代码、测试、配置等推断。
- `mixed`：部分来自声明，部分需要推断补充。

当需要推断边界时，按以下优先级收集证据：`softWare.md 缺失记录 -> 包/目录结构 -> import 与反向调用链 -> 测试 -> 配置/构建文件 -> README/架构文档`。输出中必须说明采用了哪些推断证据。

## 工作流程

### 1. 定位上下文

- 查找 `softWare.md`。
- 定位当前 SDD 工作流指定的目标成果物；如果未指定且尚无成果物，则准备创建 `xxx模块详细设计说明书` 的 ASIS 章节草案。
- 明确本次 ASIS 要支撑的需求、AR 项、功能点或变更范围。
- 根据 `softWare.md` 识别候选模块路径和模块名。
- 如果用户引用了上游 SDD 产物，定位并阅读相关内容。
- 检查受影响区域附近的构建文件、依赖清单、路由、配置、迁移、脚本和测试。
- 仓库搜索优先使用 `rg` 和 `rg --files`。

### 2. 定义分析切片

形成一段简洁的范围说明，至少包含：

- 本次支撑的需求或变更。
- `softWare.md` 记录的目标模块。
- 纳入分析的入口、文件、包、类、配置、测试、数据对象和外部交互。
- 排除在本次分析之外的模块区域，以及排除原因。
- 触发范围扩展的因素，例如共享数据、共享工具、跨模块调用、功能开关、兼容逻辑或归属不确定。

当较窄的需求相关切片已经足够支撑后续设计时，不要做完整模块深挖。

### 2.1 小模块模式

当目标只是 helper、store、parser、adapter、utility 等小模块，且影响面集中、调用点少、测试明确时，可以启用小模块模式。小模块模式允许压缩输出结构，但必须保留：

- 分析结论摘要。
- 边界来源与推断依据。
- 需求/AR 与 ASIS 证据映射。
- 关键 ASIS 事实。
- 隐藏约束与风险。
- 测试覆盖现状。
- 证据索引。

不要为了套完整模板而生成大量空表。

### 3. 建立需求/AR 覆盖映射

在目标成果物中维护 `需求/AR 与 ASIS 证据映射`，用于证明本次 ASIS 已覆盖后续 TOBE 需要的现状事实。

每条需求、AR 项、功能点或影响点至少记录：

- 编号和描述。
- 对应的现有入口、代码区域、数据对象、配置或外部交互。
- 已确认的 ASIS 事实。
- 证据编号，例如 `E1`、`E2`。
- 覆盖状态：`已覆盖`、`部分覆盖`、`待确认`、`不涉及本模块`。
- 未覆盖或待确认原因。

### 4. 建立受影响模块地图

为需求相关区域建立简明地图，覆盖相关的：

- 入口：API、Controller、Handler、Command、Job、Consumer、Listener、定时任务、CLI。
- 内部组件：领域对象、Service、Repository、Adapter、Client、Validator、Policy、工具类。
- 外部依赖：数据库表、缓存、队列、RPC/HTTP Client、SDK、文件、Feature Flag、配置中心、鉴权服务。
- 测试：单元测试、集成测试、契约测试、Fixture、迁移测试、快照测试、端到端测试、Mock、测试数据。

不要仅根据命名判断职责。重要组件必须绑定代码证据。

### 5. 逆向当前行为

对分析切片中的每个重要行为：

- 从入口追踪到副作用，梳理主调用链。
- 识别输入、输出、状态变化、持久化、事件发送和外部调用。
- 识别校验、权限、审计、指标、日志和错误映射。
- 识别事务、重试、幂等、并发、缓存、降级和超时行为。
- 识别环境差异和功能开关分支。

### 6. 挖掘隐藏约束

主动搜索会影响 TOBE 设计或 AICoding 的约束：

- 注释、TODO、FIXME、废弃标记、迁移说明、兼容分支。
- 魔法常量、默认值、硬编码枚举、哨兵值、空值处理、字段长度假设。
- Feature Flag、环境变量、配置默认值、条件注册、构建标签。
- 编码边界场景或历史行为的测试。
- 错误文案、日志文案、指标名、告警名、权限名、审计事件名。
- 数据库约束、索引、可空字段、唯一键、外键、迁移历史。
- 序列化格式、线协议兼容、API 版本、历史字段别名。
- 跨模块假设和调用顺序依赖。
- 规格漂移：设计文档、README、注释、测试和代码实现之间不一致的阈值、路径、格式、默认值、行为描述。

当受影响区域复杂、风险高或文档稀疏时，使用 `references/evidence-checklist.md` 辅助检查。

### 7. 区分事实、推断与待确认

统一使用以下标记：

- `事实`：由代码、配置、测试或已记录的 SDD 产物直接支撑。
- `推断`：由多个信号合理推出，但没有直接明示。
- `待确认`：从现有材料无法验证，但会影响后续设计或编码的重要问题。

每个重要 ASIS 结论都必须包含证据编号。证据可以是文件路径、行号、函数名、类名、配置项、路由、表名、测试名、迁移文件或命令输出摘要。能定位到行号时，优先记录到 `文件:行号`。正文中的证据必须能在目标成果物的证据索引中反查。

### 8. 更新目标成果物

更新目标成果物的 ASIS 章节时，加载 `references/asis-output-template.md`。若工作流未指定目标成果物，则默认更新或创建 `xxx模块详细设计说明书`。输出要服务后续 TOBE skill 和门禁 skill：

- 先说明本次需求或变更范围。
- 维护需求/AR 与 ASIS 证据映射。
- 具体到足以支撑工程设计。
- 严格区分 ASIS 与 TOBE。
- 除风险和待确认问题外，不提出最终实现方案。
- 范围、边界、依赖、证据和风险优先用表格表达。
- 只保留会影响详细设计或 AICoding 的待确认问题。

小范围变更可以合并章节，但至少保留：分析结论摘要、需求/AR 与 ASIS 证据映射、模块边界与本次分析切片、关键调用链或数据流、隐藏约束与风险、测试覆盖现状、待确认问题、证据索引。

## 阻塞与降级输出

如果继续分析后仍无法达到质量标准，不要用猜测补齐。必须在目标成果物中写入 `ASIS 阻塞项`，并标记 ASIS 状态为 `阻塞` 或 `部分完成`。

出现以下情况时可以阻塞：

- 无法访问必要代码或关键目录。
- `softWare.md` 与实际代码严重冲突，且无法判断模块边界。
- 缺少必要的需求、AR 或变更描述，导致无法定义分析切片。
- 关键运行时行为只存在于不可访问的配置、环境、外部系统或生产数据中。
- 上游 SDD 产物互相矛盾，导致无法判断本模块职责。

阻塞输出必须包含：阻塞原因、已完成的分析范围、不能确认的结论、继续推进需要的输入、对 TOBE/AICoding 的影响。禁止把阻塞项伪装成事实。

## 质量标准

结束前必须确认：

- 已搜索 `softWare.md`，并说明使用情况或缺失情况。
- 已明确目标需求、AR 项、功能点或计划变更。
- 已维护需求/AR 与 ASIS 证据映射。
- 已明确并分类模块边界。
- 已明确需求相关的分析切片。
- 当模块大于本次受影响范围时，已说明不在本次分析范围内的区域。
- 已检查相关入口、内部组件、外部依赖、配置、数据和测试。
- 主行为已沿真实代码路径追踪。
- 已主动搜索隐藏约束和风险。
- 已记录设计文档、README、测试与代码之间影响本次需求的规格漂移。
- 重要结论有证据。
- 正文证据编号能在证据索引中反查。
- 已区分事实、推断和待确认。
- 未将 TOBE 方案混入 ASIS 结论。

如果未达到质量标准，继续分析；如果达到阻塞条件，则按阻塞与降级输出规则写入目标成果物。
