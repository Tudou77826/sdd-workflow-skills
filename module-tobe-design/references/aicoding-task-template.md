# AICoding 实施任务拆分模板

在当前 SDD 工作流指定的正式《{AR编号}-{需求短名}-{模块名}模块详细设计说明书.md》TOBE 章节中使用本模板。任务必须能被后续 AICoding 直接执行；`.context.md` 只记录过程和证据，不承载开发任务。

## 任务拆分规则

- 每个任务只承载一个清晰目标。
- 每个任务默认只覆盖一个行为变化或一条主链路，尽量只对应一个入口点；若同时修改多个入口、多个行为变化或大范围文件，必须拆分任务或说明不能拆分的理由。
- 单个任务的主要改动文件建议控制在少量强相关文件内；跨模块、跨层、跨数据域或同时涉及接口/数据/UI/后台任务时，应优先拆分为多个任务。
- 任务之间要标明依赖关系。
- 任务必须落到具体文件、包、类、函数或明确的新建位置。
- 涉及 REST/Kafka/MQ、内部 Java 接口、数据库、配置、包图、类图或流程图变化的任务，必须关联正式说明书中的对应设计章节和编号。
- 任务必须包含验收标准和建议测试。
- 任务必须能关联到测试矩阵中的至少一个测试项，除非明确说明该任务无需测试的理由。
- 每个任务必须包含阶段性验证点，优先设计能在小步实现后立即运行的单元测试、契约测试或局部集成测试。
- 每个任务必须包含最小验证集，明确到建议测试文件、测试用例或命令；不能只写“补充单测”。
- 如果任务引用的 ASIS 结论是 `推断`、`待确认`、低置信度或阻塞相关，任务不得直接进入编码，除非任务目标就是补证据、补测试或消除阻塞。
- 高风险任务必须说明风险来源和验证方式。
- 不要使用“优化”“完善”“处理一下”这类不可执行表述。

## 验收标准写法

验收标准应描述可验证行为，例如：

- 当输入满足某条件时，目标接口返回指定状态或字段。
- 当外部依赖失败时，系统按指定错误码、重试或降级策略处理。
- 当配置开关关闭时，保持 ASIS 行为不变。
- 当迁移完成后，新旧数据读写兼容。
- 指定单元测试、集成测试或契约测试通过。
