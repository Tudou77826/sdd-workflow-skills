# 模块详细设计门禁检查清单

对当前 SDD 工作流指定的目标成果物执行门禁时使用本清单。根据实际技术栈和变更范围选择适用项，但阻断项不能跳过。

## 0. 门禁场景

- 判断当前是 `真实门禁` 还是 `演练门禁`。
- 演练门禁需分别记录 `skill 有效性结论` 和 `真实 AICoding 准入结论`。
- 区分 `代码门禁` 与 `文档门禁`，允许演练模式中出现“代码门禁通过，文档门禁存在缺口”的中间态。

## 1. 输入完整性

- 需求/AR/功能点/变更范围明确。
- 目标模块明确，模块边界来自 `softWare.md` 或有合理推断说明。
- ASIS 章节存在。
- TOBE 章节存在。
- 需求/AR 与 ASIS 证据映射存在。
- 证据索引存在。
- 上游设计对齐存在。
- 测试设计存在。
- 风险、兼容、迁移、回滚设计存在或说明不适用。
- AICoding 任务拆分存在。

## 2. 代码证据

- 关键 ASIS 结论引用证据编号。
- 证据编号可在证据索引中反查。
- 证据覆盖代码、配置、测试、数据、迁移、外部依赖和隐藏约束中与本次需求相关的部分。
- `推断`、`待确认` 没有被 TOBE 当成事实直接使用。
- 无证据的重要判断已被标记为待确认或阻塞。

## 2.1 ASIS 倾向采样

至少抽查 3 个 ASIS 结论；少于 3 个则全部抽查。优先抽查：

- 高风险结论。
- 跨模块结论。
- 推断或待确认结论。
- 低证据结论。
- 规格漂移。
- 测试缺口。
- 高频/批量路径。

每个采样项记录采样理由、证据复核结果和门禁影响。

## 3. 需求与设计一致性

- 每条需求/AR 都有处理结论。
- 本模块承担与不承担的内容清楚。
- TOBE 满足需求和功能设计。
- TOBE 符合 AR 职责划分。
- TOBE 符合模块影响性分析和模块间交互设计。
- 没有越界承担其他模块职责。
- 没有遗漏本模块应承担的功能点。
- ASIS 中发现的规格漂移已有 TOBE 处理结论，或被列为待确认/阻塞项。

## 4. 工程可实现性

- 新增、修改、复用、废弃组件分别说明。
- 组件设计能落到文件、包、类、函数或明确新建位置。
- 主流程、异常流程、边界场景清楚。
- 接口、数据、配置、开关、权限、审计、日志、指标按需明确。
- 事务、并发、幂等、重试、超时、降级、补偿、数据一致性按需处理。
- 兼容、迁移、灰度、回滚可执行。
- AICoding 任务能直接编码。

## 4.1 TOBE 严格探索

- 架构反证：检查循环依赖、跨层调用、重复规则、共享状态扩散、旁路写入、隐式耦合。
- 替代方案对比：检查是否存在更小、更贴合现有架构的方案。
- 主流程追踪：入口到副作用逐步走通。
- 失败路径追踪：至少覆盖外部依赖失败、输入异常、部分成功。
- 日志追踪：主流程可追溯，高频路径避免海量日志。
- 安全探索：输入、路径、命令、权限、敏感信息、审计边界明确。
- 性能探索：批量规模、I/O、内存、复杂度、缓存或降级有合理上限。
- 可测试性探索：任务能分阶段验证。

## 5. 可测试性

- 测试覆盖主流程、异常流程、边界场景。
- 测试覆盖兼容、迁移、回归、契约或跨模块交互中的适用项。
- 测试落到现有测试文件或明确新增测试位置。
- 每个 AICoding 任务有验收标准和建议测试。
- 存在测试矩阵，能从需求/AR 追踪到 TOBE 设计点、任务和测试项。
- 不存在无法验证的关键设计结论。

## 6. 风险控制

- 高风险变更已识别。
- 风险触发条件、影响范围、缓解措施、回滚方案清楚。
- 性能、安全、权限、数据一致性、兼容性、运维风险已按需覆盖。
- 日志与可观测性风险已覆盖，包括海量日志、缺少关联字段、错误路径不可追踪、敏感信息泄露。
- 阻塞项和待确认问题真实暴露。

## 7. 文档质量

- ASIS 与 TOBE 分离清楚。
- 事实、推断、待确认和设计决策分离清楚。
- 没有“完善逻辑”“优化代码”“按需处理”等不可执行表述。
- 表格编号、证据编号、需求/AR 编号、任务编号稳定一致。
- 后续 AICoding 不需要重新猜测设计意图。
