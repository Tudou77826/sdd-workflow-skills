# 模块详细设计正式说明书四链门禁检查清单

本清单是辅助检查，不替代门禁结论。门禁结论以正式《模块详细设计说明书》是否能作为唯一开发依据，以及四条链是否闭合为准。`.context.md` 只用于证据和过程复核，不能弥补正式说明书缺失的开发指导内容。

## 1. 证据链

- 需求/AR 能追溯到 ASIS 结论和证据编号。
- ASIS 证据编号能在证据索引中反查。
- ASIS 结论状态保持为事实 / 推断 / 待确认 / 阻塞相关。
- 推断、待确认、低置信度或阻塞相关结论没有被 TOBE 当成事实。
- 关键 TOBE 决策引用 ASIS 证据。
- 已做 ASIS 倾向采样；高风险、跨模块、低置信度、规格漂移或核心链路场景已风险加抽。

## 2. 边界链

- 模块边界来自 `software_architecture.md` 或明确推断。
- 本模块承担和不承担的内容清楚。
- TOBE 没有越界承担其他模块职责。
- 跨模块交互符合模块影响性分析和模块间交互设计。
- 没有引入循环依赖、跨层调用、外部模型穿透、共享状态扩散、旁路写入或隐式耦合。
- 跨边界改动已说明防腐层、DTO/模型翻译、依赖方向和禁止穿透内容。

## 3. 执行链

- 每个关键 TOBE 决策落到工程位置。
- 被触发的包/模块结构变化已在正式说明书中提供包图、结构变化表和依赖方向。
- 被触发的类/接口变化已在正式说明书中提供类图、类/接口清单、关键方法和设计约束。
- 被触发的业务流程变化已在正式说明书中提供主流程图；异常、回滚、异步或状态变化按需展开。
- REST 接口变化已在正式说明书中提供 method、path、认证/权限、请求、响应、错误码和兼容性。
- Kafka/MQ 接口变化已在正式说明书中提供 topic、producer/consumer、消息结构、key、幂等、重试、死信和兼容策略。
- 内部 Java 接口/抽象变化已在正式说明书中提供调用方、实现方、方法签名、入参出参、异常语义和扩展约束。
- 数据库变化已在正式说明书中提供表/字段/索引、约束、默认值、迁移/回填、兼容和回滚。
- 配置/开关变化已在正式说明书中提供配置项、默认值、环境差异、灰度策略和关闭后 ASIS 行为。
- 字段、DTO、接口、配置项、事件名、状态值、错误码、topic、path、时间单位和 JSON/序列化名称在摘要、图、表、工程落点、任务、测试和风险章节中前后一致。
- AICoding 任务粒度合适，能直接编码。
- 任务有主要改动区域、依赖、验收标准和最小验证集。
- 测试矩阵能从需求/AR 追踪到 ASIS 证据、TOBE 决策、任务、测试项和断言方式。
- 不存在必须等全部实现完成后才能验证的关键设计。

## 4. 风险链

只检查被触发的风险专题；未触发专题要有不适用说明。

- 架构防腐化风险有防护策略和验证方式。
- 安全风险按普通/增强强度处理。
- 性能风险按普通/增强强度处理。
- 日志可观测性能串联主流程、定位错误路径，并避免海量日志和敏感信息泄露。
- 兼容、迁移、回滚策略可执行。
- 并发、事务、幂等、重试、超时、降级、补偿按需处理。

## 5. 阻断分类

- `EVIDENCE`：证据不足或不确定性被伪装成事实。
- `BOUNDARY`：边界、职责、交互或防腐化不成立。
- `EXECUTION`：任务不可执行、不可验证或粒度失控。
- `RISK`：触发风险没有缓解、验证或回退。
