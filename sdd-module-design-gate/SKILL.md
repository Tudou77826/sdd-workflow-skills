# SDD 模块设计门禁

## 阶段规则

本 skill 已内置门禁阶段必须遵守的规则，可独立检查正式说明书；完整共享方法论由 `$sdd-module-detailed-design-flow` 在串联流程时读取。仅当需要处理跨阶段冲突、输出模式争议或编排规则时，才参考 skill 组根目录的 `sdd-module-design-method.md`。

门禁不检查“章节是否写满”，而检查正式说明书能否作为唯一开发依据，以及四条链是否闭合：

- 证据链：需求/AR -> ASIS 证据 -> TOBE 决策。
- 边界链：模块边界 -> 职责分配 -> 交互约束。
- 执行链：TOBE 决策 -> AICoding 任务 -> 最小验证集。
- 风险链：触发风险 -> 缓解设计 -> 验证/回退。

## 成果物协作规则

- 正式成果物命名为 `{AR编号}-{需求短名}-{模块名}模块详细设计说明书.md`，例如 `AR07758-支持快速退款接口相关功能-支付审计模块详细设计说明书.md`。
- 过程上下文命名为 `{AR编号}-{需求短名}-{模块名}模块详细设计说明书.context.md`。
- `xxx模块详细设计说明书` 只是占位名；同一轮 SDD 工作流中必须解析为同一个正式说明书和同名前缀 context 文件。
- 本 skill 在正式说明书中的写入边界是门禁结论、四链摘要、阻断问题、整改清单和阻塞说明。
- 本 skill 在 `.context.md` 中的写入边界是 ASIS 采样记录、TOBE 反证探索、详细四链检查、多轮复检记录和审计材料。
- 本 skill 可以读取 ASIS、TOBE、任务、测试矩阵和风险章节，但不得直接覆盖其正文。
- 只有工作流、用户或成果物拆分规则明确要求时，门禁结果才写入独立门禁成果物；否则优先写回正式说明书的门禁章节。
- 门禁引用的需求/AR 编号、TOBE 决策编号和任务编号必须来自正式说明书；ASIS 证据编号必须能在同名前缀 `.context.md` 的证据索引中反查。
- 如果某项开发必需内容只存在于 `.context.md`，正式说明书门禁必须判定为不通过，问题类型为 `EXECUTION` 或对应链路问题。

## 门禁结论

门禁结论只能是：

- `通过`：四条链均通过，阻断问题为 0，可进入 AICoding。
- `不通过`：存在必须整改的问题，需回到 ASIS、TOBE 或上游设计对齐。
- `阻塞`：缺少必要输入、代码、上游决策，或连续返工仍因同一外部缺口无法推进。

演练门禁必须分别输出 `skill 有效性结论` 和 `真实 AICoding 准入结论`，不得把演练通过等同于真实准入。

## 工作流程

### 1. 定位双文件成果物

- 读取当前 SDD 工作流指定的正式说明书；未指定时使用同一轮流程内统一的 `{AR编号}-{需求短名}-{模块名}模块详细设计说明书.md` 占位。
- 读取同名前缀 `.context.md` 作为证据和过程复核材料，但不得把 context 中独有的开发信息视为正式说明书已满足。
- 从正式说明书读取 ASIS、TOBE、需求/AR 映射、AICoding 任务、测试矩阵、待确认问题和阻塞项；从同名前缀 `.context.md` 读取证据索引和复核材料。
- 识别运行模式：真实 / 演练。

### 2. 检查证据链

判断需求/AR 是否能追溯到 ASIS 证据和 TOBE 决策：

- 关键 ASIS 结论是否有证据编号。
- 证据编号是否能在证据索引中反查。
- ASIS 结论状态是否保留为事实/推断/待确认/阻塞相关。
- 推断、待确认、低置信度或阻塞相关结论是否被 TOBE 错当成事实。
- TOBE 关键决策是否引用 ASIS 证据。

必须进行 ASIS 倾向采样。至少抽查 3 个 ASIS 结论；少于 3 个则全查。高风险、大规模、跨模块、低置信度、边界冲突、规格漂移或核心链路影响时，必须风险加抽。

### 3. 检查边界链

判断模块边界和职责是否约束住设计：

- 模块边界是否来自 `software_architecture.md` 或明确推断。
- 本模块承担和不承担的内容是否清楚。
- TOBE 是否越界承担其他模块职责。
- 跨模块交互是否符合模块影响性分析和模块间交互设计。
- 是否引入循环依赖、跨层调用、外部模型穿透、共享状态扩散、旁路写入或隐式耦合。

存在跨边界改动时，必须检查防腐层、DTO/模型翻译、依赖方向和禁止穿透内容。

### 4. 检查执行链

判断 TOBE 是否能交给 AICoding 执行：

- 每个关键 TOBE 决策是否落到工程位置。
- 正式说明书是否包含被触发的包图、类图、业务流程图、对外接口、内部接口、数据库变化、配置变化等开发视图。
- REST 接口变化是否包含 method、path、权限、请求、响应、错误码和兼容性。
- Kafka/MQ 接口变化是否包含 topic、producer/consumer、消息结构、key、幂等、重试、死信和兼容策略。
- 内部 Java 接口或抽象变化是否包含调用方、实现方、方法签名、入参出参、异常语义和扩展约束。
- 数据库变化是否包含表/字段/索引、约束、默认值、迁移/回填、兼容和回滚。
- AICoding 任务是否粒度合适，能直接编码。
- 任务是否有主要改动区域、依赖、验收标准和最小验证集。
- 测试矩阵是否能从需求/AR 追踪到 ASIS 证据、TOBE 决策、任务、测试项和断言方式。
- 是否存在无法阶段性验证的关键设计。
- 字段、DTO、接口、配置项、事件名、状态值、错误码、topic、path、时间单位和 JSON/序列化名称是否在摘要、类图、流程图、接口表、数据设计、工程落点、任务、测试和风险章节中保持一致。

任务过大、改动位置不清、测试不可执行时，执行链不通过。

### 5. 检查风险链

只检查被触发的风险专题，不要求所有专项表都填满：

- 架构防腐化。
- 安全增强。
- 性能增强。
- 日志可观测性。
- 兼容、迁移与回滚。
- 并发、事务与幂等。

每个触发风险都必须有缓解设计、验证方式和必要回退策略。未触发专题应有简短不适用说明。

### 5.1 检查前置问询

- 影响模块边界、核心方案、数据/权限/兼容策略、性能目标、验收阈值或任务拆分的待确认项，是否已在 ASIS 或 TOBE 阶段触发问询。
- 如果没有问询条件，成果物是否标记为 `需前置确认` 并阻止相关决策定稿。
- 如果待确认项可能推翻当前设计，不得给出真实 AICoding 准入通过结论。

### 6. TOBE 严格探索

门禁要主动探索 TOBE 正确性，不只读表格：

- 架构反证：是否可构造出边界腐化或依赖方向错误的反例。
- 替代方案对比：是否存在更小、更贴合现有架构的方案。
- 主流程追踪：入口到副作用是否走通。
- 失败路径追踪：至少覆盖输入异常、外部依赖失败、部分成功或回退场景中的适用项。
- 可测试性追踪：是否能按任务分阶段验证。
- 契约一致性追踪：抽取关键字段、DTO、接口、配置项或事件，沿摘要、设计变量表、图、表、任务和测试逐项核对类型、单位、默认值、序列化名称和兼容语义。

探索发现的问题必须进入阻断问题或非阻断改进项。

### 7. 输出门禁结果

加载 `references/gate-result-template.md`，输出四链门禁结果、ASIS 采样、TOBE 探索、阻断问题、整改清单和复检记录。

阻断问题只分四类：

- `EVIDENCE`：证据不足或不确定性被伪装成事实。
- `BOUNDARY`：边界、职责、交互或防腐化不成立。
- `EXECUTION`：任务不可执行、不可验证或粒度失控。
- `RISK`：触发风险没有缓解、验证或回退。

## 多轮复检规则

门禁不通过时：

1. 生成整改清单。
2. 判断每个整改项应回到 ASIS、TOBE、上游设计对齐还是模块间交互设计。
3. 如果当前 agent 能补齐，回到对应阶段更新正式说明书和同名前缀 `.context.md` 后重新门禁。
4. 如果不能补齐，标记阻塞并说明需要的输入。
5. 复检时保留轮次记录，说明完成项、待闭合问题和整改要求。

## 质量标准

结束前确认：

- 门禁结论是 `通过`、`不通过` 或 `阻塞` 之一。
- 四条链均有明确结论。
- 阻断问题与非阻断改进项分离。
- 每个阻断问题都有明确返工阶段和整改要求。
- 通过时明确说明可进入 AICoding。
- 不通过或阻塞时明确说明不可进入 AICoding，或仅允许哪些有限范围进入 AICoding。
