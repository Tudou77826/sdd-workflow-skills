# SDD 模块设计门禁

## 目标

检查当前 SDD 工作流指定的模块详细设计类成果物是否具备进入 AICoding 的质量。门禁不是简单评审意见，而是质量闭环：发现阻断项后要明确返工阶段、整改内容和下一轮检查重点，直到通过或确认阻塞。

## 成果物协作规则

遵循当前 SDD 工作流或编排 skill 指定的成果物归属规则。对于本 skill，默认检查模块详细设计类成果物；如果工作流没有另行指定，则使用 `xxx模块详细设计说明书`。

- 优先读取用户或工作流提供的目标成果物。
- 不在没有依据的情况下另起孤立门禁文档；门禁结果应写入目标成果物的门禁章节，或按工作流指定写入独立门禁成果物。
- 门禁引用的 ASIS 证据编号、需求/AR 编号、TOBE 决策编号、任务编号必须来自目标成果物。
- 如果门禁发现 ASIS 或 TOBE 需要返工，整改项必须能定位到具体章节、表格、编号或缺失证据。

## 输入

可以接受以下任意组合：

- 当前 SDD 工作流指定的目标成果物路径或当前草稿内容；未指定时默认为 `xxx模块详细设计说明书`。
- ASIS 章节、TOBE 章节、证据索引、需求/AR 映射。
- 需求分析、功能设计、AR 设计。
- 模块影响性分析、模块间交互设计。
- 代码仓路径和目标模块信息。
- 团队模板、质量规则、编码规范或发布要求。

## 门禁结论

门禁结论只能是：

- `通过`：阻断问题为 0，设计可进入 AICoding。
- `不通过`：存在必须整改的问题，需回到 ASIS、TOBE 或上游设计对齐。
- `阻塞`：连续返工仍无法推进，或缺少必要输入/代码/上游决策，当前无法给出可执行详设。

非阻断改进项可以存在，但不得影响 AICoding 开始。

## 门禁场景

先识别当前门禁场景：

- `真实门禁`：用于判断是否可进入真实 AICoding。
- `演练门禁`：用于验证 skill、样例需求或 forward-test，不代表真实需求已经具备编码准入条件。

演练门禁必须分别输出：

- `skill 有效性结论`：这些 skill 是否能引导完成 ASIS、TOBE 和门禁流程。
- `真实 AICoding 准入结论`：如果要进入真实编码，还缺少哪些正式输入或成果物。

缺少正式需求、AR、`softWare.md` 或目标成果物路径，在演练门禁中通常是文档元信息缺口；只有当这些缺口阻断样例流程验证时，才判定 skill 验证失败。

## 工作流程

### 1. 定位目标成果物

- 定位当前 SDD 工作流指定的目标成果物；未指定时默认定位 `xxx模块详细设计说明书`。
- 读取 ASIS、TOBE、证据索引、需求/AR 映射、待确认问题、阻塞项、AICoding 任务。
- 如果目标成果物缺失，先判断是否能从上下文恢复；不能恢复则输出阻塞。

### 2. 检查输入完整性

检查是否具备：

- 本次需求/AR/功能点/变更范围。
- 目标模块和模块边界。
- ASIS 章节。
- TOBE 章节。
- 需求/AR 与 ASIS 证据映射。
- ASIS 证据索引。
- 上游设计对齐。
- 测试设计。
- 风险、兼容、迁移、回滚设计。
- AICoding 任务拆分。

### 3. 检查代码证据

- 关键 ASIS 结论是否引用证据编号。
- 证据编号是否能在证据索引中反查。
- 证据是否覆盖代码、配置、测试、数据、迁移、外部依赖和隐藏约束中与本次需求相关的部分。
- 是否存在无证据的重要判断。
- ASIS 的 `推断` 和 `待确认` 是否被正确处理，没有被 TOBE 当成事实直接使用。

### 3.2 ASIS 倾向采样

门禁不能只顺序阅读 ASIS。必须进行有倾向的采样来判断 ASIS 质量，优先抽查：

- 高风险结论：影响数据、安全、权限、兼容、迁移、回滚、性能的 ASIS 结论。
- 跨模块结论：涉及外部模块、共享表、共享配置、事件、RPC/API 的结论。
- 推断结论：标记为 `推断`，但被 TOBE 使用的结论。
- 待确认结论：标记为 `待确认`，但未进入阻塞或风险的结论。
- 低证据结论：只有文档证据、没有代码/测试证据的重要结论。
- 规格漂移：文档与代码不一致的阈值、路径、默认值、行为描述。
- 测试缺口：影响本次设计但没有测试覆盖的现状行为。
- 高频/批量路径：可能带来海量日志、性能或内存风险的路径。

每轮门禁至少抽查 3 个 ASIS 结论；如果 ASIS 结论少于 3 个，则全部抽查。抽查结果必须写明“采样项、采样理由、证据复核结果、是否影响门禁”。

### 3.1 区分代码门禁与文档门禁

门禁结果中应区分：

- `代码门禁`：代码证据、测试、实现路径、工程可行性是否成立。
- `文档门禁`：成果物路径、正式需求/AR、`softWare.md`、编号索引、章节完整性是否满足工作流要求。

在演练模式中，可以出现 `代码门禁通过，文档门禁存在缺口` 的中间态。该中间态说明 skill 可执行，但真实进入 AICoding 前仍需补齐正式文档元信息。

### 4. 检查需求与设计一致性

- 每条需求/AR 是否有处理结论。
- 本模块承担与不承担的内容是否清楚。
- TOBE 是否满足需求和功能设计。
- TOBE 是否符合 AR 职责划分。
- TOBE 是否符合模块影响性分析和模块间交互设计。
- 是否存在越界承担其他模块职责。
- 是否遗漏本模块应承担的功能点。

### 5. 检查工程可实现性

- 新增、修改、复用、废弃组件是否分别说明。
- 组件设计是否能落到文件、包、类、函数或明确新建位置。
- 主流程、异常流程、边界场景是否清楚。
- 接口、数据、配置、开关、权限、审计、日志、指标是否按需明确。
- 事务、并发、幂等、重试、超时、降级、补偿、数据一致性是否按需处理。
- 兼容、迁移、灰度和回滚是否可执行。
- AICoding 任务是否能直接编码。

### 5.1 TOBE 严格探索

门禁必须主动探索 TOBE 的正确性，而不是只检查是否填了表。至少执行以下探索：

- 架构反证：该设计是否引入循环依赖、跨层调用、重复业务规则、共享状态扩散、旁路写入或隐式耦合。
- 替代方案对比：是否存在更小、更贴合现有架构的方案；若有，当前方案为什么仍合理。
- 主流程追踪：从入口到副作用逐步走通，确认每一步有组件承担。
- 失败路径追踪：至少选择一个外部依赖失败、一个输入异常、一个部分成功场景进行推演。
- 日志追踪：主流程是否能通过有限日志串起来；高频路径是否避免逐条海量日志。
- 安全探索：输入、路径、命令、权限、敏感信息、审计是否有明确边界。
- 性能探索：批量规模、I/O、内存、复杂度、缓存或降级是否有合理上限。
- 可测试性探索：是否能按任务分阶段验证，而不是等完整实现后才验证。

探索发现的问题必须进入阻断问题或非阻断改进项，不能只写在说明文字里。

### 6. 检查可测试性

- 测试设计是否覆盖主流程、异常流程、边界场景。
- 是否包含兼容、迁移、回归、契约或跨模块测试。
- 测试是否落到现有测试文件或明确新增测试位置。
- 每个 AICoding 任务是否有验收标准和建议测试。
- 是否存在无法验证的设计结论。

### 7. 检查风险控制

- 是否识别高风险变更。
- 是否说明风险触发条件、影响范围、缓解措施和回滚方案。
- 是否覆盖性能、安全、权限、数据一致性、兼容性、运维风险。
- 是否覆盖日志与可观测性风险，包括海量日志、缺少关联字段、错误路径不可追踪、敏感信息泄露。
- 阻塞项和待确认问题是否真实暴露，没有被弱化成普通备注。

### 8. 检查文档质量

- ASIS 与 TOBE 是否分离清楚。
- 事实、推断、待确认和设计决策是否分离清楚。
- 是否存在泛泛表述，例如“完善逻辑”“优化代码”“按需处理”。
- 表格编号、证据编号、需求/AR 编号、任务编号是否稳定一致。
- 文档是否足以让后续 AICoding 不重新猜测设计意图。

### 9. 输出门禁结果

加载 `references/gate-result-template.md`，将门禁结果写入目标成果物的门禁章节，或按工作流指定输出独立门禁成果物。

每个阻断问题必须包含：

- 问题编号。
- 严重级别。
- 所属门禁类别。
- 关联章节或编号。
- 具体问题。
- 为什么阻断 AICoding。
- 需要回到的阶段：ASIS / TOBE / 上游设计对齐 / 模块间交互设计。
- 整改要求。
- 下一轮复检重点。

## 多轮复检规则

门禁不通过时，不要只停留在评论层面：

1. 生成整改清单。
2. 判断每个整改项应回到 ASIS、TOBE 还是上游设计对齐。
3. 如果当前 agent 能补齐，直接补齐目标成果物后重新门禁。
4. 如果不能补齐，标记阻塞并说明需要的输入。
5. 重新门禁时保留轮次记录，说明本轮修复了哪些问题，剩余哪些问题。

通过条件：

- 阻断问题为 0。
- 代码证据门禁通过。
- ASIS 倾向采样未发现阻断问题。
- 设计一致性门禁通过。
- 可实现性门禁通过。
- TOBE 严格探索未发现阻断问题。
- 可测试性门禁通过。
- 风险控制没有阻断项。

## 阻塞规则

出现以下情况时可以标记 `阻塞`：

- 缺少必要代码、目标成果物或上游 SDD 产物。
- 模块边界、模块间交互或 AR 职责存在无法自行解决的冲突。
- ASIS 关键证据不可访问，且 TOBE 依赖该证据。
- 关键业务规则、兼容要求、数据迁移或权限策略需要用户/负责人决策。
- 连续多轮整改仍因同一外部输入缺失无法通过。

阻塞时必须输出：阻塞原因、已完成检查范围、无法通过的门禁项、需要补充的输入、当前是否允许有限范围 AICoding。

## 质量标准

结束前必须确认：

- 已检查所有适用门禁类别。
- 门禁结论是 `通过`、`不通过` 或 `阻塞` 之一。
- 阻断问题与非阻断改进项分离。
- 每个阻断问题都有明确返工阶段和整改要求。
- 通过时明确说明可进入 AICoding。
- 不通过或阻塞时明确说明不可进入 AICoding，或仅允许哪些有限范围进入 AICoding。
