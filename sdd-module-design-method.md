# SDD 模块详设核心方法论

## 核心哲学

以代码证据确认现状，以模块边界约束设计，以最小充分决策支撑 AICoding，以可验证任务闭合风险。

## 成果物双文件规则

模块详细设计阶段产出一组同名前缀文件：

- 正式成果物：`{AR编号}-{需求短名}-{模块名}模块详细设计说明书.md`。
- 过程上下文：`{AR编号}-{需求短名}-{模块名}模块详细设计说明书.context.md`。

例如：

- `AR07758-支持快速退款接口相关功能-支付审计模块详细设计说明书.md`
- `AR07758-支持快速退款接口相关功能-支付审计模块详细设计说明书.context.md`

正式《模块详细设计说明书》是指导开发的最后一道文档，也是后续 AICoding 的唯一开发依据。它必须同时满足人类快速审计和 AI 直接开发，承载所有会影响编码实现、接口契约、包/类设计、数据库设计、配置、调用流程、验证方式、风险控制和回退策略的内容。

`.context.md` 只作为生成正式说明书的过程文档，用于记录证据、检索过程、边界确认、详细 ASIS、TOBE 推导、替代方案、门禁采样和复检记录。它不参与后续指导开发。禁止将指导开发所必需的设计内容仅写入 `.context.md`。

如果某项内容会影响开发实现、接口、类设计、数据库、业务流程、任务拆分、验证方式或风险控制，必须进入正式说明书；`.context.md` 可以保留其证据和推导过程。

## 成果物写入职责

- ASIS 阶段只写同名前缀 `.context.md`，记录 ASIS 证据、检索过程、边界确认、关键结论、待确认问题和阻塞项；不得创建、编辑或覆盖正式说明书。
- TOBE 阶段是正式说明书唯一写入者，负责基于 `.context.md` 中的 ASIS 证据生成或更新正式说明书，并把 TOBE 推导、替代方案和追踪矩阵写回 `.context.md`。
- Gate 阶段只读正式说明书和 `.context.md` 做准入检查，门禁结论、阻断问题、整改清单、采样和复检记录写入 `.context.md` 或在当前会话返回；不得编辑正式说明书正文。
- 正式说明书中的 ASIS 内容只能由 TOBE 阶段以“ASIS 证据引用”形式写入，不展开 ASIS 检索过程。

## 正式说明书标准大纲

正式说明书必须遵循以下一级大纲。未涉及的一级章节保留标题并写明“不涉及，原因：...”，不要生成大量空表。

1. 需求背景 / 当前 AR 描述
2. 外部依赖
3. 对外接口
4. 整体方案
5. 数据库 / 表设计
6. 受影响模块
7. 模块交互设计
8. 模块详细方案
9. 附录 / 三方件约束

第 8 章是 AICoding 的核心开发依据，必须按需求点展开，不能只用总表替代正文。每个需求点至少展开：

- 需求描述。
- ASIS 证据引用。
- TOBE 设计说明。
- 类 / 接口 / 数据结构变化。
- 处理流程。
- 异常、回退与边界场景。
- 配置、开关与环境差异。
- 日志与可观测性。
- 测试用例。
- AICoding 任务。

## 架构文档前置条件

ASIS 必须且只能使用当前工作区 `.sdd/software_architecture.md` 作为模块边界来源。该文件缺失、不可读、目标模块缺失、声明含糊或与用户指定模块冲突时，中断 ASIS，并停止后续 TOBE 和门禁流程。仓库根目录、子目录中的 `software_architecture.md`，以及 `softWare.md`、`software.md`、README 或其他架构说明，均不能替代该前置条件。

检查该前置条件时必须优先读取直接路径，例如 `.sdd/software_architecture.md`。如果使用搜索命令辅助定位隐藏目录，必须使用能包含隐藏目录的形式，例如 `rg --files -uu .sdd`；不得用裸 `rg --files` 的空结果判定 `.sdd/software_architecture.md` 缺失。

## 协作写入规则

更新成果物前，必须读取目标文件当前内容，只替换本阶段负责的章节或明确占位区域。多 agent 或多阶段并行时，同一成果物只设一个最终合并者，其他参与者提供补丁、章节草案或 review 意见，避免覆盖其他阶段或人工补充内容。

如果目标仓库 `.sdd/` 中已有未跟踪、未提交或疑似他人正在编辑的成果物，不得复用同名文件做演练或试跑。优先选择新的演练编号、外部 `sdd-trial-output/` 目录，或只在当前会话输出草案；确需更新已有文件时，先读取当前内容并只替换本阶段负责的明确章节。

## 七条原则

1. 证据优先：ASIS 只确认事实、推断和不确定性。没有证据的关键结论不能进入 TOBE 定稿。
2. 边界优先：TOBE 必须受模块职责和交互边界约束，不能为了局部需求破坏模块边界。
3. 最小充分：正式说明书不是过程日志，但必须完整指导开发。小变更轻量，大变更展开，高风险专项增强；不得把开发必需信息转移到 `.context.md`。
4. 任务可执行：每个设计决策最终要落到可编码任务、改动位置、验收标准和最小验证集。
5. 风险显性化：不确定、阻塞、架构腐化、安全、性能、日志、兼容等风险必须显性化；只有被触发时才展开专项分析。
6. 契约一致：字段、DTO、接口、配置项、事件名、状态值、错误码、topic、path、时间单位和 JSON/序列化名称必须在正式说明书内单点定义并跨章节复用；方案调整时同步更新摘要、图、表、任务、测试和风险章节。
7. 契约保真：用户输入、上游设计、模块交互、ASIS context 或已有正式说明书中已经明确给出的接口定义、方法/函数签名、入参出参、DTO/字段、JSON/序列化名称、错误码、状态值、Topic、Path、配置项和数据结构，必须原样或结构化等价进入正式说明书和 AICoding 任务；不得被摘要、改名或省略。

## 四条闭合链

门禁和详设都围绕四条链判断，不以章节数量判断质量。

| 闭合链 | 判断问题 | 典型阻断 |
|---|---|---|
| 证据链 | 需求/AR 是否能追溯到 ASIS 证据和明确结论状态 | 关键事实无证据、推断被当事实、低置信度未暴露 |
| 边界链 | 模块边界、职责分配和交互约束是否清楚 | 越界承担、跨层调用、循环依赖、外部模型穿透 |
| 执行链 | TOBE 决策是否落到 AICoding 任务和最小验证集 | 任务过大、改动位置不清、无法阶段性验证 |
| 风险链 | 被触发的风险是否有缓解、验证和回退 | 安全/性能/兼容/日志/并发等风险未闭合 |

## 三档输出模式

| 模式 | 适用场景 | 输出要求 |
|---|---|---|
| 轻量 | 单入口 bug、helper/adapter/parser 小改、配置/文案/测试补齐，且无高风险触发 | 正式说明书保留开发必需的决策、落点、任务和验证；不涉及章节写明原因，不生成空表；第 8 章仍需按需求点展开 |
| 标准 | 一般功能改动，通常已拆分到单模块内，涉及多个组件但边界清晰 | 正式说明书按标准大纲写清被触发的外部依赖、接口、流程、数据、模块交互和详细方案 |
| 增强 | 数据迁移、权限/隐私/支付/密钥、高频/批处理/核心链路、明确 SLA、门禁返工、ASIS 低置信度、复杂跨模块交互 | 正式说明书补齐所有被触发的设计视图、风险专题、验证和回退；context 保留更完整过程证据 |

## 正式说明书必备开发视图

TOBE 涉及变化时，正式说明书必须按触发项输出对应设计视图。未涉及时写明“不涉及”，不要生成空表。

| 变化类型 | 正式说明书必须包含 |
|---|---|
| 包/模块结构变化 | 包图、结构变化表、依赖方向和边界说明 |
| 类/接口变化 | 类图、类/接口/抽象类清单、完整方法/函数签名、入参出参、职责和约束 |
| 业务流程变化 | 主流程图；涉及异常、回滚、异步、状态变化时补充异常/回退流程 |
| 对外 REST 接口变化 | Method、Path、认证/权限、请求、响应、错误码、兼容性 |
| Kafka/MQ 接口变化 | Topic、Producer/Consumer、消息结构、Key、幂等、重试、死信、兼容策略 |
| 内部调用或 Java 抽象变化 | 调用方、实现方、完整方法/函数签名、入参出参、异常语义、扩展约束 |
| 数据库变化 | 表/字段/索引、约束、默认值、迁移/回填、兼容策略、回滚策略 |
| 配置/开关变化 | 配置项、默认值、环境差异、灰度策略、关闭后的 ASIS 行为 |

正式说明书中的图和表只覆盖本次需求相关切片，不画全系统。它们要能让 AI 开发直接据此实现，也要让人类评审快速审计边界、接口、数据和风险。

## 风险专题

风险专题按触发条件展开，不作为所有需求的固定大表。成果物中使用“风险专题设计”“专项风险分析”等正式表述，避免使用实现味过重或自造感强的词。

- 架构防腐化：跨模块、跨层、依赖方向、外部模型、公共接口、共享状态变化。
- 安全增强：支付、隐私、凭证/密钥、权限边界、跨租户数据、外部输入、文件路径、命令执行、审计合规、AI/LLM 上下文污染。
- 性能增强：高频调用、批处理、大数据量、同步核心链路、启动路径、核心交易链路、明确 SLA、资源瓶颈、缓存/降级策略变化。
- 日志可观测性：主流程追踪、错误路径定位、日志密度、敏感信息脱敏、指标/告警。
- 兼容/迁移/回滚：接口签名、数据结构、配置默认值、迁移回填、灰度和回退。
- 并发/事务/幂等：状态变更、重复请求、重试、超时、补偿、数据一致性。

## 原子 skill 职责

- ASIS：证明现状是什么，哪些不确定。
- TOBE：说明基于现状和边界，决定怎么改。
- Gate：判断这些决定能不能安全交给 AICoding。
- Flow：只做临时串联和演练，不定义新标准。

## 前置问询原则

待确认问题越早暴露越好。凡是会影响模块边界、职责归属、核心方案、数据/权限/兼容策略、性能目标、验收阈值或 AICoding 任务拆分的待确认项，应在发现时立即向用户或上游负责人问询；若当前执行环境不能直接问询，必须在成果物中标记为“需前置确认”，并阻止依赖该问题的 TOBE 决策定稿。

## 图示原则

当需求涉及多组件协作、跨模块交互、状态变化、异步/回调、写后读、回滚或异常路径时，ASIS 或 TOBE 至少提供一个能指导开发的流程图或时序图。优先使用 Mermaid，并保持图只覆盖关键链路，不为简单单函数修改强制画图。
