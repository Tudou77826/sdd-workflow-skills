# SDD 模块详设轻量流程

## 方法论

本 skill 负责完整读取并应用 skill 组根目录的共享方法论 `sdd-module-design-method.md`，再串联 ASIS -> TOBE -> Gate。若安装目录与开发仓库不同，优先使用当前 skill 组根目录下的同名方法论文档。

本 skill 只是薄编排，用于串联 ASIS -> TOBE -> Gate，不定义新的详设标准。

## 目标

在没有正式编排器时，临时串联模块 ASIS、TOBE 和门禁三个原子 skill，验证它们能围绕当前 SDD 工作流指定的正式说明书和同名前缀 `.context.md` 协作：ASIS 只写 context，TOBE 是正式说明书唯一写入者，Gate 只写 context 或返回结论。

正式 SDD 工作流存在时，优先服从正式编排器。本 skill 只做薄编排，不定义新的设计标准。

## 演练模式

当用户要求“练手”“验证 skill”“forward-test”“不改仓库”或没有正式上游 SDD 产物时，启用演练模式。

- 明确标记当前需求为 `样例需求` 或 `演练需求`。
- ASIS、TOBE、Gate 摘要区都必须标记 `运行模式=演练`；演练模式不得在后续流转中被改写为 `真实`。摘要区对应：ASIS context 的 `C1. ASIS 分析结论摘要`，TOBE 正式说明书的第 1 章或变更记录摘要，Gate context 的门禁结论摘要。
- 允许输出纯叙述式验证结果，而不是强制落盘到仓库内成果物。
- 演练模式需要落盘且用户未指定路径时，默认写入目标仓库 `.sdd/`，命名为 `AR-SDD-TRIAL-NNN-{演练需求短名}-{模块名}模块详细设计说明书.md` 和同名前缀 `.context.md`；`NNN` 选择当前 `.sdd/` 中未使用的最小三位序号。
- 如果用户要求“不改仓库”或目标仓库不适合写入，默认写入当前工作区外部目录 `sdd-trial-output/`，保持同样命名规则，并在最终结果中明确这些文件不是真实 SDD 准入产物。
- 不修改目标代码仓，除非用户明确要求。
- 门禁结论需区分 `skill 有效性结论` 和 `真实 AICoding 准入结论`。
- 缺少正式需求、AR 或正式说明书路径时，不应直接判定 skill 失败；应记录为真实流程的文档元信息缺口。
- 缺少 `.sdd/software_architecture.md`、文件不可读或目标模块缺失时，ASIS 必须中断；演练模式也不得用其他架构文档或代码结构替代模块边界输入。
- 检查 `.sdd/software_architecture.md` 必须先查直接路径；如果使用文件枚举辅助定位隐藏目录，使用 `rg --files -uu .sdd` 或等价命令，不得用裸 `rg --files` 的空结果判定 `.sdd` 缺失。

## 使用的原子 skill

按顺序使用：

1. `$sdd-module-asis-analysis`
2. `$sdd-module-tobe-design`
3. `$sdd-module-design-gate`

## 输入

可以接受：

- 当前 SDD 工作流指定的正式说明书和同名前缀 `.context.md`；未指定时按 `{AR编号}-{需求短名}-{模块名}模块详细设计说明书.md` 规则命名。
- 代码仓路径。
- 目标模块。
- 需求、功能点、AR、模块影响性分析、模块间交互设计。
- 用户关注的接口、数据、配置、风险、测试或验收标准。

## 流程

### 1. 确认双文件成果物

- 如果工作流指定了正式说明书，使用该成果物，并用同名前缀推导 `.context.md`，除非工作流另行指定。
- 如果未指定，默认按 `{AR编号}-{需求短名}-{模块名}模块详细设计说明书.md` 推导正式说明书路径和同名前缀 `.context.md`；正式说明书只由 TOBE 阶段创建或更新。
- 正式说明书是最终交付和唯一开发依据；`.context.md` 记录 ASIS 证据、TOBE 推导和门禁过程。
- 如果是演练模式且用户要求不改仓库，可以在当前会话或外部输出目录产出验证报告；若需要双文件演练产物，按演练模式命名规则创建，不复用真实 AR 编号。
- 保留已有成果物内容；ASIS 只更新 `.context.md` 的 ASIS 章节，TOBE 更新正式说明书并补充 `.context.md` 推导章节，Gate 只更新 `.context.md` 的门禁章节或返回结论。
- 每次写入成果物前，先读取目标文件当前内容；只替换本阶段负责的章节或明确标记的占位区域。不得整文件重写导致其他阶段、其他 agent 或人工补充内容丢失。
- 多 agent 或多阶段并行试用时，同一成果物只允许一个写入者负责最终合并；其他参与者输出补丁、章节草案或 review 意见，由合并者读当前文件后合入。

### 2. 执行 ASIS

使用 `$sdd-module-asis-analysis`：

- 根据 `.sdd/software_architecture.md` 识别模块边界。
- `.sdd/software_architecture.md` 是必要输入；缺失、不可读或目标模块缺失时中断 ASIS，并停止后续 TOBE 和门禁流程。
- `.sdd` 是隐藏目录时，先直接读取 `.sdd/software_architecture.md`；不要让通用文件搜索忽略隐藏目录而造成误判。
- 围绕本次需求相关切片分析 ASIS。
- `.context.md` 写入详细 ASIS 证据、检索过程、边界确认记录、调用链展开、证据索引、阻塞项或待确认问题。
- ASIS 阶段不得创建或编辑正式说明书。

如果 ASIS 阻塞且无法继续 TOBE，停止流程并输出阻塞原因。

### 3. 执行 TOBE

使用 `$sdd-module-tobe-design`：

- 基于 ASIS 证据和上游 SDD 产物设计 TOBE。
- 对字段、DTO、接口、配置项、事件名、状态值、错误码、topic、path 和时间单位建立单点定义，并在正式说明书各章节保持一致。
- 正式说明书按 9 个一级章节写入：需求背景/当前 AR 描述、外部依赖、对外接口、整体方案、数据库/表设计、受影响模块、模块交互设计、模块详细方案、附录/三方件约束。
- 第 8 章“模块详细方案”必须按需求点展开，包含需求描述、ASIS 证据引用、TOBE 设计说明、类/接口/数据结构变化、处理流程、异常回退边界场景、配置开关环境差异、日志可观测性、测试用例和 AICoding 任务。
- `.context.md` 写入 TOBE 推导依据、替代方案、反证记录和完整追踪矩阵。

如果 TOBE 阻塞且无法进入门禁，停止流程并输出阻塞原因。

### 4. 执行门禁

使用 `$sdd-module-design-gate`：

- 检查正式说明书是否可进入 AICoding。
- 以正式说明书为准检查是否可进入 AICoding；`.context.md` 只用于过程复核，不能弥补正式说明书缺失的开发指导内容。
- 检查关键字段、DTO、接口、配置项、事件名、状态值、错误码、topic、path、时间单位和 JSON/序列化名称是否跨章节一致。
- `.context.md` 输出通过、不通过或阻塞结论、四链摘要、阻断问题、整改清单、证据追踪链、ASIS 倾向采样、TOBE 严格探索、日志与可观测性专项、测试矩阵复核和复检记录；也可以在当前会话直接返回门禁结论。
- Gate 阶段不得编辑正式说明书正文。
- 不通过时生成整改清单和返工阶段。

### 5. 返工复检

如果门禁不通过：

- 返工阶段为 ASIS 时，回到 `$sdd-module-asis-analysis` 补证据或修正 ASIS。
- 返工阶段为 TOBE 时，回到 `$sdd-module-tobe-design` 修正设计。
- 返工阶段为上游设计对齐或模块间交互设计时，记录需要外部输入；如果当前上下文足够，由 TOBE 阶段补齐正式说明书中的对齐说明。
- 返工后重新运行 `$sdd-module-design-gate`。

最多持续到门禁通过或出现无法自行解决的阻塞。不要用猜测消除阻塞项。

## 输出

最终输出应说明：

- 正式说明书。
- 配套 `.context.md`。
- ASIS 状态。
- TOBE 状态。
- 门禁结论。
- ASIS 倾向采样是否发现问题。
- TOBE 严格探索是否发现问题。
- 日志与可观测性、测试矩阵是否通过复核。
- 是否可进入 AICoding。
- 若不可进入，列出阻断问题和下一步需要补充的输入。
