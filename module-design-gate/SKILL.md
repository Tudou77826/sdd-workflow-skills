# SDD 模块设计门禁

## 阶段规则

本 skill 自身定义门禁阶段必须遵守的规则，可独立检查正式说明书；无论由何种入口调用，均以本 skill 的规则作为 Gate 阶段唯一执行依据。

## 运行方式

本 skill 应由当前主 Agent 直接执行，不要通过 SubAgent 启动；遇到需要用户或上游确认的问题，应在当前会话及时问询，避免 SubAgent 的运转模式导致问询滞后。

门禁不检查“章节是否写满”，而检查正式说明书能否作为唯一开发依据，以及四条链是否闭合：

- 证据链：需求/AR -> ASIS 证据 -> TOBE 决策。
- 边界链：模块边界 -> 职责分配 -> 交互约束。
- 执行链：TOBE 决策 -> AICoding 任务 -> 最小验证集。
- 风险链：触发风险 -> 缓解设计 -> 验证/回退。

## 成果物协作规则

- 正式成果物命名为 `{AR编号}-{需求短名}-{模块名}模块详细设计说明书.md`，例如 `AR07758-支持快速退款接口相关功能-支付审计模块详细设计说明书.md`。
- 过程上下文命名为 `{AR编号}-{需求短名}-{模块名}模块详细设计说明书.context.md`。
- `xxx模块详细设计说明书` 只是占位名；同一轮 SDD 工作流中必须解析为同一个正式说明书和同名前缀 context 文件。
- 本 skill 不得创建、编辑或覆盖正式说明书中的任何章节。
- 本 skill 在 `.context.md` 中的写入边界是门禁结论、四链摘要、阻断问题、整改清单、阻塞说明、ASIS 采样记录、TOBE 反证探索、详细四链检查、多轮复检记录和审计材料。
- 本 skill 可以读取 ASIS、TOBE、任务、测试矩阵和风险章节，但不得直接覆盖其正文。
- 只有工作流、用户或成果物拆分规则明确要求时，门禁结果才写入独立门禁成果物；否则优先写入同名前缀 `.context.md` 的门禁过程章节，或在当前会话返回。
- 门禁引用的需求/AR 编号、TOBE 决策编号和任务编号必须来自正式说明书；ASIS 证据编号必须能在同名前缀 `.context.md` 的证据索引中反查。
- 如果某项开发必需内容只存在于 `.context.md`，正式说明书门禁必须判定为不通过，问题类型为 `EXECUTION` 或对应链路问题。
- 正式说明书必须是面向用户、评审和开发的标准交付件；若正式说明书包含 ASIS 检索过程、证据编号表、TOBE 推导过程、反证记录、门禁采样、追踪矩阵等过程性内容，或必须阅读 `.context.md` 才能理解设计结论，门禁必须判定为不通过，问题类型为 `EXECUTION`。

## 门禁结论

门禁结论只能是：

- `通过`：四条链均通过，阻断问题为 0，可进入 AICoding。
- `不通过`：存在必须整改的问题，需回到 ASIS、TOBE 或上游设计对齐。
- `阻塞`：缺少必要输入、代码、上游决策，或连续返工仍因同一外部缺口无法推进。

门禁还必须给出明确的门禁建议，用于指导下一轮重试：

- `可进入 AICoding`：所有适用维度达标，且阻断问题为 0。
- `回 ASIS 补证据后重试`：证据充分性未达标，或 ASIS 证据不能支撑 TOBE。
- `回 TOBE 补设计后重试`：决策定稿性、契约完整性、工程可执行性、验证闭环性或风险处置性未达标。
- `先做上游/用户确认`：核心方案、模块边界、接口归属、数据归属、验收阈值或跨模块职责仍需外部确认。
- `拆分范围后有限进入 AICoding`：未达标维度不影响某个独立子范围，且正式说明书已明确隔离该子范围的任务、边界和验证方式。
- `阻塞，缺少必要输入`：无法完成门禁判断所需的正式说明书、context、代码、架构边界或上游决策。

除 `拆分范围后有限进入 AICoding` 的受限场景外，只要存在任一适用维度未达标，门禁结论不得为 `通过`。

## 维度化门禁建议

门禁必须按以下维度输出 `达标 / 未达标 / 不适用`，并为每个未达标维度给出发现、影响、建议返工阶段和下一轮重试通过条件。维度检查不替代四链检查；它把四链问题转换成可执行的返工建议。

| 维度 | 关注点 | 未达标时默认建议 |
|---|---|---|
| 证据充分性 | 需求、ASIS 结论、证据索引是否足以支撑 TOBE | 回 ASIS 补证据后重试 |
| 边界清晰性 | 模块边界、职责、不承担范围、跨模块关系是否清楚 | 回 ASIS 或上游架构确认 |
| 决策定稿性 | 正式说明书是否仍存在影响实现的建议、可选、如有需要、待定或待确认项 | 回 TOBE 补设计后重试 |
| 契约完整性 | 新增/修改的函数、接口、DTO、配置、错误码、路径规则等是否有名称、位置、签名/字段、类型、调用方、错误语义和兼容策略 | 回 TOBE 补设计后重试 |
| 工程可执行性 | AICoding 任务是否落到具体文件、函数、类型、调用点、依赖顺序和完成判定 | 回 TOBE 补设计后重试 |
| 验证闭环性 | 每个关键设计目标是否有测试或验证步骤，关键失败路径是否可验证 | 回 TOBE 补设计后重试；若缺现状测试证据则回 ASIS |
| 风险处置性 | 触发的安全、兼容、回滚、性能、并发、观测风险是否有缓解、验证和回退 | 回 TOBE 补设计后重试或上游确认 |

维度未达标的常见判定：

- 影响实现的“建议 / 可选 / 如有需要 / 推荐 / 尽量 / 是否需要 / 待定”没有被明确标为非阻断改进项。
- 只写“抽一个 helper”“增加校验”“优化错误文案”“实现上述方案”，但没有写清契约、调用边界或完成判定。
- 关键目标或风险缓解没有对应测试、断言、命令或人工验证步骤。
- `.context.md` 中存在影响实现的待确认项，但正式说明书或门禁结论没有把它转成阻断问题。
- 关键开发信息只存在于 `.context.md`，正式说明书自身不能指导编码。

## 工作流程

### 1. 定位双文件成果物

- 读取当前 SDD 工作流指定的正式说明书；若正式说明书路径未指定或文件不存在，门禁状态为 `阻塞`，不得用占位文件名替代正式说明书进行准入判断。
- 读取同名前缀 `.context.md` 作为证据和过程复核材料，但不得把 context 中独有的开发信息视为正式说明书已满足。
- 从正式说明书读取需求/AR、TOBE 设计、AICoding 任务、测试矩阵、待确认问题和阻塞项；从同名前缀 `.context.md` 读取 ASIS 结论、证据索引、历史门禁结果、上一轮阻断问题、整改清单、复检记录和复核材料。
- 如果正式说明书或 `.context.md` 中存在上一轮未关闭的阻断问题、整改项、待确认项或阻塞项，必须纳入本轮门禁输入，不得只检查当前正式说明书快照。

### 2. 检查证据链

判断需求/AR 是否能追溯到 ASIS 证据和 TOBE 决策：

- 关键 ASIS 结论是否有证据编号。
- 证据编号是否能在证据索引中反查。
- ASIS 结论状态是否在 `.context.md` 中保留为事实/推断/待确认/阻塞相关。
- 推断、待确认、低置信度或阻塞相关结论是否被 TOBE 错当成事实。
- TOBE 关键决策是否能在 `.context.md` 中反查 ASIS 证据。
- 正式说明书是否用用户可读的现状约束摘要承接关键 ASIS 约束，而不是输出内部证据编号或检索过程。

必须进行 ASIS 倾向采样。至少抽查 3 个 ASIS 结论；少于 3 个则全查。高风险、大规模、跨模块、低置信度、边界冲突、规格漂移或核心链路影响时，必须风险加抽。

### 3. 检查边界链

判断模块边界和职责是否约束住设计：

- 模块边界是否来自当前工作区 `.sdd/software_architecture.md` 的直接声明；不得接受代码结构、README、根目录 `software_architecture.md`、`softWare.md`、用户口述或“明确推断”作为模块边界来源。
- 本模块承担和不承担的内容是否清楚。
- TOBE 是否越界承担其他模块职责。
- 跨模块交互是否符合模块影响性分析和模块间交互设计。
- 是否引入循环依赖、跨层调用、外部模型穿透、共享状态扩散、旁路写入或隐式耦合。

存在跨边界改动时，必须检查防腐层、DTO/模型翻译、依赖方向和禁止穿透内容。

### 4. 检查执行链

判断 TOBE 是否能交给 AICoding 执行：

- 每个关键 TOBE 决策是否落到工程位置。
- 正式说明书是否包含被触发的包图、类图、业务流程图、对外接口、内部接口、数据库变化、配置变化等开发视图。
- 正式说明书是否作为标准交付件自洽可读，不包含 ASIS 检索过程、证据编号表、TOBE 推导过程、反证记录、门禁采样、追踪矩阵等过程性内容。
- 用户输入、上游设计、模块交互、ASIS context 或已有正式说明书中已经明确给出的接口定义、方法/函数签名、入参出参、DTO/字段、JSON/序列化名称、错误码、状态值、Topic、Path、配置项和数据结构，是否在正式说明书中原样或结构化等价保留。
- REST 接口变化是否包含 method、path、权限、请求、响应、错误码和兼容性。
- Kafka/MQ 接口变化是否包含 topic、producer/consumer、消息结构、key、幂等、重试、死信和兼容策略。
- 内部 Java 接口或抽象变化是否包含调用方、实现方、方法签名、入参出参、异常语义和扩展约束。
- 数据库变化是否包含表/字段/索引、约束、默认值、迁移/回填、兼容和回滚。
- AICoding 任务是否粒度合适，能直接编码。
- 任务是否有主要改动区域、依赖、验收标准和最小验证集。
- 任务是否说明具体文件、函数/类型/调用点、变更动作、任务依赖和完成判定。
- 新增或修改的函数、接口、DTO、配置、错误码、路径规则等是否给出名称、所在文件/包、导出性或可见性、签名/字段/类型、调用方、错误语义和兼容策略。
- 正式说明书中是否仍有影响实现的“建议 / 可选 / 如有需要 / 推荐 / 尽量 / 是否需要 / 待定”；若有，是否已明确标为非阻断改进项且不影响当前 AICoding 范围。
- 正式说明书中的测试矩阵是否能从需求/AR 追踪到 TOBE 决策、任务、测试项和断言方式；ASIS 证据追踪保留在 `.context.md`。
- 每个关键设计目标是否都有测试、断言、命令或人工验证步骤；关键失败路径不能只写成可选测试。
- 是否存在无法阶段性验证的关键设计。
- 字段、DTO、接口、配置项、事件名、状态值、错误码、topic、path、时间单位和 JSON/序列化名称是否在摘要、类图、流程图、接口表、数据设计、工程落点、任务、测试和风险章节中保持一致。

任务过大、改动位置不清、测试不可执行时，执行链不通过。

### 5. 检查风险链

只检查被触发的风险专题，不要求所有专项表都填满：

- 架构防腐化。
- 安全增强。
- 性能增强。
- 日志可观测性。
- 兼容、迁移与回滚。
- 并发、事务与幂等。

每个触发风险都必须有缓解设计、验证方式和必要回退策略。未触发专题应有简短不适用说明。

### 5.1 检查前置问询

- 影响模块边界、核心方案、数据/权限/兼容策略、性能目标、验收阈值或任务拆分的待确认项，是否已在 ASIS 或 TOBE 阶段触发问询。
- 如果没有问询条件，成果物是否标记为 `需前置确认` 并阻止相关决策定稿。
- 如果待确认项可能推翻当前设计，不得给出真实 AICoding 准入通过结论。

### 5.2 检查遗留问题闭环

- 读取上一轮门禁阻断问题、整改清单、复检记录、正式说明书待确认问题、正式说明书阻塞项、ASIS 待确认问题、ASIS 阻塞项和 TOBE 部分完成/阻塞说明。
- 遗留问题必须保留原始编号、来源轮次和来源章节；本轮可以增加本轮跟踪编号，但不得用新编号覆盖或掩盖原始问题编号。
- 每个遗留问题必须有明确闭环状态：`已关闭`、`转为非阻断`、`仍阻断` 或 `阻塞`。
- 标记为 `已关闭` 的问题必须能反查到正式说明书章节、`.context.md` 证据/推导记录、ASIS/TOBE 更新记录或外部输入确认；只有口头说明或“已处理”字样不得视为闭环。
- 标记为 `转为非阻断` 的问题必须说明不影响 AICoding 准入的理由、剩余风险和后续跟踪位置。
- 标记为 `仍阻断` 或 `阻塞` 的问题必须继续进入本轮阻断问题或阻塞说明，并明确不可进入 AICoding 或仅允许的有限范围。
- 上一轮遗留问题未逐项复核时，本轮门禁不得给出 `通过` 结论。

### 6. TOBE 严格探索

门禁要主动探索 TOBE 正确性，不只读表格：

- 模拟 AICoding 任务拆解：从正式说明书抽取每个任务，判断是否能在不读取 `.context.md`、不追问用户的情况下开始编码；不能则归类为 `EXECUTION` 阻断。
- 架构反证：是否可构造出边界腐化或依赖方向错误的反例。
- 替代方案对比：是否存在更小、更贴合现有架构的方案。
- 主流程追踪：入口到副作用是否走通。
- 失败路径追踪：至少覆盖输入异常、外部依赖失败、部分成功或回退场景中的适用项。
- 可测试性追踪：是否能按任务分阶段验证。
- 契约一致性追踪：抽取关键字段、DTO、接口、配置项或事件，沿摘要、设计变量表、图、表、任务和测试逐项核对类型、单位、默认值、序列化名称和兼容语义。
- 契约保真追踪：抽取上文已经明确给出的接口定义或方法签名，核对正式说明书是否保留完整签名、每个入参出参、字段名、JSON 名称、错误码和来源；若正式说明书只保留自然语言概要，必须判定执行链不通过。

探索发现的问题必须进入阻断问题或非阻断改进项。

### 7. 输出门禁结果

加载 `<skill-dir>/references/gate-result-template.md`，输出门禁结论、门禁建议、维度化门禁检查、四链门禁结果、遗留问题闭环复核、ASIS 采样、TOBE 探索、阻断问题、整改清单和复检记录。门禁结果写入 `.context.md` 或作为当前会话结果返回，不写入正式说明书正文。

阻断问题只分四类：

- `EVIDENCE`：证据不足或不确定性被伪装成事实。
- `BOUNDARY`：边界、职责、交互或防腐化不成立。
- `EXECUTION`：任务不可执行、不可验证、粒度失控，或正式说明书丢失上文已经明确给出的接口契约、方法签名、入参出参、DTO/字段等开发必需信息。
- `RISK`：触发风险没有缓解、验证或回退。

每个阻断问题必须关联至少一个未达标维度，并写明下一轮重试通过条件。

## 多轮复检规则

门禁不通过时：

1. 生成整改清单。
2. 判断每个整改项应回到 ASIS、TOBE、上游设计对齐还是模块间交互设计。
3. 如果当前 agent 能补齐，回到对应阶段更新对应成果物后重新门禁：ASIS 只更新 `.context.md`，TOBE 更新正式说明书并补充 `.context.md` 推导过程。
4. 如果不能补齐，标记阻塞并说明需要的输入。
5. 复检时保留轮次记录，说明完成项、待闭合问题和整改要求。
6. 下一轮门禁必须先复核上一轮遗留问题闭环情况；未闭环的问题不得丢失、重编号后掩盖或从阻断问题中静默移除。

## 质量标准

结束前确认：

- 门禁结论是 `通过`、`不通过` 或 `阻塞` 之一。
- 已输出门禁建议，且建议与未达标维度一致。
- 已按证据充分性、边界清晰性、决策定稿性、契约完整性、工程可执行性、验证闭环性、风险处置性逐项给出 `达标 / 未达标 / 不适用`。
- 四条链均有明确结论。
- 上一轮遗留问题、整改项、待确认项和阻塞项均有闭环状态和证据；未逐项复核时不得通过。
- 阻断问题与非阻断改进项分离。
- 每个阻断问题都有明确返工阶段和整改要求。
- 每个未达标维度都有下一轮重试通过条件。
- 通过时明确说明可进入 AICoding。
- 不通过或阻塞时明确说明不可进入 AICoding，或仅允许哪些有限范围进入 AICoding。
