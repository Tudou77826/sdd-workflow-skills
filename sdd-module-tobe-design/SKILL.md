# SDD 模块 TOBE 设计

## 目标

在当前 SDD 工作流指定的模块详细设计类成果物中，基于 ASIS 事实和上游 SDD 产物，补充模块 TOBE 详细设计，使后续门禁和 AICoding 能够直接使用。

TOBE 设计要回答：为了实现本次需求，目标模块应该如何调整现有职责、组件、流程、接口、数据、配置、测试和实施任务。

## 成果物协作规则

遵循当前 SDD 工作流或编排 skill 指定的成果物归属规则。对于本 skill，默认协作目标是模块详细设计类成果物；如果工作流没有另行指定，则使用 `xxx模块详细设计说明书`。

- 优先读取并更新用户或工作流提供的目标成果物。
- 不在没有依据的情况下另起孤立 TOBE 文档；只有当工作流、用户或成果物拆分规则要求时，才拆分输出。
- 保留已有 ASIS、证据索引、需求/AR 映射、待确认问题和阻塞项。
- TOBE 中引用的 ASIS 事实必须使用 ASIS 章节已有证据编号，例如 `E1`、`E2`。
- 新增的 TOBE 决策、任务、风险、测试编号应保持稳定，供后续门禁引用。
- 如果修改了 ASIS 相关判断，必须同步更新证据索引或待确认问题，不得只在 TOBE 中覆盖旧结论。

## 输入

可以接受以下任意组合：

- 当前 SDD 工作流指定的模块详细设计类成果物路径或当前草稿内容；未指定时默认为 `xxx模块详细设计说明书`。
- 已完成或部分完成的 ASIS 章节。
- 需求分析、功能设计、AR 设计。
- 模块影响性分析、模块间交互设计。
- 目标模块名称、目录、包路径、服务、类、API、Job、Consumer 或功能区域。
- 编码约束，例如技术栈、代码风格、兼容策略、测试要求、发布策略。
- 用户关注的接口、数据表、消息主题、配置项、迁移、风险或验收标准。

如果目标成果物或 ASIS 章节缺失，先判断能否从代码和上游产物回补必要 ASIS。若不能回补，则按“阻塞与回补规则”处理。

## 设计原则

- 以需求/AR 为目标，以 ASIS 证据为约束。
- 只设计本次需求相关的模块内 TOBE，不做无关重构。
- 保持模块职责边界，不把其他模块职责塞进本模块。
- 明确架构影响，避免为了局部需求引入跨层调用、循环依赖、共享状态膨胀、重复抽象、旁路写入等架构腐化。
- 每个重要 TOBE 决策都要说明对应的 ASIS 事实、需求/AR 来源和设计理由。
- 对新增、修改、复用、废弃分别说明，不混在一起。
- 设计必须能拆成 AICoding 可执行任务。
- 设计必须能分阶段验证，不能等到大段实现完成后才第一次验证。
- 不确定项不能伪装成设计结论；要进入待确认问题、风险或阻塞项。

如果当前是演练模式、forward-test 或样例需求，TOBE 只能标记为 `验证样例`，不得声称已经满足真实 AICoding 准入。真实编码前仍需补齐正式需求/AR、目标成果物归属和必要上游设计。

## 工作流程

### 1. 读取目标成果物

- 定位当前 SDD 工作流指定的目标成果物；如果未指定，则默认定位或创建 `xxx模块详细设计说明书`。
- 读取 ASIS 章节、需求/AR 与 ASIS 证据映射、证据索引、待确认问题、ASIS 阻塞项。
- 读取已有 TOBE 章节；如果存在，判断是补充、修正还是重写。
- 如果用户提供了新上游产物，先与目标成果物中的已有内容对齐。

### 2. 检查 TOBE 前置条件

确认以下内容是否足够支撑设计：

- 本次需求、AR 项、功能点或变更范围明确。
- 模块边界明确，且来自 `softWare.md` 或有合理推断说明。
- ASIS 覆盖了本次 TOBE 需要修改或依赖的关键入口、流程、数据、配置、测试和外部交互。
- 需求/AR 与 ASIS 证据映射中没有影响设计的未覆盖项。
- ASIS 阻塞项不会阻断 TOBE 设计。

如果前置条件不足，先回补 ASIS；如果无法回补，按阻塞与回补规则处理。

### 3. 对齐上游设计

在目标成果物中维护 `上游设计对齐`：

- 本模块承担哪些需求/AR。
- 哪些需求/AR 不由本模块承担。
- 本模块与其他模块的交互边界。
- 本模块新增、修改、复用、废弃的能力。
- 与模块影响性分析、模块间交互设计不一致的地方及处理结论。

### 3.1 设计架构影响

明确本次 TOBE 对架构的影响：

- 是否改变模块职责、分层边界、调用方向、数据归属或状态所有权。
- 是否新增跨模块依赖、共享结构、公共接口、后台任务、缓存、配置或事件。
- 是否可能引入架构腐化，例如循环依赖、跨层直接调用、重复业务规则、全局状态扩散、旁路写入、隐式耦合。
- 是否需要保留现有架构模式，或新增轻量抽象来隔离变化。
- 是否需要同步更新模块影响性分析、模块间交互设计或 `softWare.md`。

### 4. 设计 TOBE 职责

基于 ASIS 职责和上游设计，明确：

- 保留的现有职责。
- 调整的职责。
- 新增的职责。
- 废弃或迁移的职责。
- 明确不属于本模块的职责。

每条职责变化必须引用需求/AR 编号和 ASIS 证据编号。

### 5. 设计组件级方案

分别设计：

- 新增组件：名称、位置、职责、输入输出、依赖、创建理由。
- 修改组件：当前位置、修改点、修改前行为、修改后行为、影响范围。
- 复用组件：复用方式、复用限制、为什么不新增。
- 废弃或迁移组件：废弃原因、迁移方式、兼容期、清理条件。

组件设计要落到可定位的文件、包、类、函数或明确的新文件位置。暂时无法定位时，标记为 `待确认`。

### 6. 设计流程

覆盖以下流程，按本次需求选择适用项：

- 主流程。
- 异常流程。
- 边界场景。
- 跨模块交互流程。
- 数据一致性流程。
- 灰度、降级、回滚或兼容流程。

流程设计必须说明入口、关键判断、调用链、状态变化、副作用、错误处理和观测点。

### 7. 设计接口、数据和配置

按需明确：

- API/RPC/消息/事件/Job/CLI 的入参、出参、错误码、幂等键、版本策略。
- 数据表、字段、索引、迁移、回填、清理、兼容读写策略。
- 缓存 Key、TTL、失效策略。
- 配置项、Feature Flag、灰度开关、默认值、环境差异。
- 权限、安全、审计、日志、指标、告警。

如果涉及破坏性变更，必须给出兼容、迁移和回滚方案。

接口兼容策略必须显式说明：是否修改现有函数/API 签名，是否新增结构体或只读查询接口，旧调用方是否保持不变，错误返回语义是否变化。

### 8. 设计工程保护

明确异常、事务、并发、幂等、重试、超时、限流、降级、补偿和数据一致性策略。若某项不适用，说明不适用原因。

同时必须显式设计以下质量属性：

- 日志与可观测性：主流程必须可追溯；错误路径必须可定位；避免在循环、批处理、高频路径输出海量日志；说明日志级别、关键字段、采样/聚合策略和脱敏要求。
- 安全性：输入校验、路径/命令/权限边界、敏感信息脱敏、最小权限、审计要求。
- 性能：复杂度、I/O 次数、内存占用、批量规模、缓存策略、超限降级和压力风险。
- 可测试性：每个阶段都能被单元测试或小范围集成测试验证；避免必须写完大块代码后才能验证。

### 9. 设计测试

测试设计必须覆盖：

- 主流程测试。
- 异常流程测试。
- 边界场景测试。
- 兼容性或迁移测试。
- 跨模块交互或契约测试。
- 回归测试。

每条测试建议尽量落到现有测试文件或建议新增测试文件，并说明验收标准。

必须输出测试矩阵，将需求/AR、ASIS 证据、TOBE 设计点、AICoding 任务和测试项关联起来，便于门禁自动复核。

### 10. 拆分 AICoding 任务

将 TOBE 设计拆成可执行任务。每个任务至少包含：

- 任务编号。
- 修改目标。
- 涉及文件或代码区域。
- 实现要点。
- 依赖任务。
- 验收标准。
- 建议测试。
- 风险等级。
- 阶段性验证点。

任务必须能被后续 AICoding 直接执行，避免“优化代码”“完善逻辑”这类不可落地表述。

### 11. 更新目标成果物

加载 `references/tobe-output-template.md`，将 TOBE 内容写入目标成果物的 TOBE 章节。若工作流未指定目标成果物，则默认更新或创建 `xxx模块详细设计说明书`。小范围变更可以合并章节，但至少保留：上游设计对齐、TOBE 职责、组件设计、流程设计、测试设计、AICoding 任务拆分、风险与回滚、待确认问题。

## 阻塞与回补规则

如果 TOBE 设计依赖的 ASIS 事实缺失，先回到 ASIS 分析补证据。可以在目标成果物中追加 `TOBE 设计阻塞项`，但不能用猜测补齐关键设计。

出现以下情况时标记阻塞：

- ASIS 阻塞项直接影响本次 TOBE 设计。
- 需求/AR 与 ASIS 证据映射存在关键未覆盖项。
- 上游设计互相矛盾，导致无法判断本模块职责。
- 模块间交互设计缺失或冲突，导致接口、事件、数据归属无法确定。
- 关键兼容、迁移、权限或数据一致性问题无法从现有材料判断。

阻塞输出必须包含：阻塞原因、受影响的 TOBE 章节、需要回补的 ASIS 或上游输入、当前可继续设计的范围、对 AICoding 的影响。

## 质量标准

结束前必须确认：

- TOBE 写入的是当前 SDD 工作流指定的目标成果物；未指定时写入默认的 `xxx模块详细设计说明书`。
- 每条重要设计决策引用了需求/AR 来源和 ASIS 证据编号。
- 设计没有越过模块职责边界。
- 新增、修改、复用、废弃的组件分别说明清楚。
- 主流程、异常流程、边界场景和关键工程保护已覆盖。
- 接口、数据、配置、兼容、迁移、回滚按需明确。
- 规格漂移已有处理结论，或被列为待确认/阻塞项。
- 测试设计能验证本次 TOBE 行为。
- 已输出测试矩阵，能从需求追踪到测试。
- AICoding 任务能落到具体文件或明确代码区域。
- 未决问题和阻塞项没有被伪装成设计结论。

如果未达到质量标准，继续补充设计；如果达到阻塞条件，则按阻塞与回补规则写入目标成果物。
