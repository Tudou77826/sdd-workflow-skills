# SDD 模块 TOBE 设计

## 方法论

先阅读共享方法论：`C:\Users\15802\.codex\skills\sdd-module-design-method.md`。

本 skill 只回答一个问题：基于 ASIS 事实和模块边界，本模块决定怎么改，才能让 AICoding 不重新猜测设计意图。

TOBE 质量不由章节数量决定，而由四条链是否闭合决定：

- 证据链：需求/AR -> ASIS 证据 -> TOBE 决策。
- 边界链：模块边界 -> 职责分配 -> 交互约束。
- 执行链：TOBE 决策 -> AICoding 任务 -> 最小验证集。
- 风险链：触发风险 -> 缓解设计 -> 验证/回退。

## 成果物协作规则

- `xxx模块详细设计说明书` 只是占位名；同一轮 SDD 工作流中必须解析为同一个目标成果物。
- 本 skill 的写入边界是 TOBE 章节、TOBE 变更记录、设计决策、工程落点、风险专题设计、AICoding 任务、测试矩阵、TOBE 待确认问题和 TOBE 阻塞项。
- 本 skill 可以读取 ASIS、门禁和上游设计作为输入，但不得覆盖 ASIS 事实或门禁结论。
- TOBE 中引用的 ASIS 事实必须使用 ASIS 章节已有证据编号。
- ASIS 结论状态必须贯穿到 TOBE：引用 `推断`、`待确认`、低置信度或阻塞相关结论时，必须继续标记状态，不得降级成事实。
- 如果 ASIS 事实需要修正，标记“需回到 ASIS 修正”，不要在 TOBE 中覆盖旧事实。

## 输入

可以接受：

- 当前 SDD 工作流指定的目标成果物路径或草稿内容。
- 已完成或部分完成的 ASIS 章节。
- 需求分析、功能设计、AR 设计。
- 模块影响性分析、模块间交互设计。
- 目标模块名称、目录、包路径、服务、类、API、Job、Consumer 或功能区域。
- 编码约束、测试要求、发布策略、验收标准。

如果目标成果物或 ASIS 章节缺失，先判断能否从代码和上游产物回补必要 ASIS；不能回补时按阻塞规则处理。

## 输出模式

根据共享方法论选择 `轻量 / 标准 / 增强` 输出模式。

轻量模式适用于小范围变更、helper/adapter/parser 小改、配置/文案/测试补齐、单入口低风险 bug 修复。轻量模式只减少空表和重复章节，不降低质量门禁。

轻量模式至少保留：

- 运行模式、TOBE 状态、输出模式和关键结论。
- 模块边界与本模块承担/不承担内容。
- 影响本次设计的 ASIS 事实、证据编号和结论状态。
- TOBE 设计决策和工程落点。
- AICoding 任务、验收标准和最小验证集。
- 风险专题的不适用说明或触发专题的简要分析。

出现跨模块、数据迁移、权限/隐私/支付/密钥、高频/批处理/核心链路、明确 SLA、门禁返工、ASIS 低置信度、规格漂移严重等情况时，升级为标准或增强模式。

## 工作流程

### 1. 读取目标成果物和方法论

- 读取共享方法论。
- 定位当前 SDD 工作流指定的目标成果物；未指定时使用同一轮流程内统一的 `xxx模块详细设计说明书` 占位。
- 读取 ASIS 章节、需求/AR 与 ASIS 证据映射、证据索引、待确认问题、ASIS 阻塞项。
- 读取上游设计、模块影响性分析、模块间交互设计和已有门禁整改项。

### 2. 初步对齐设计范围

先粗略回答：

- 本次需求/AR 中哪些由本模块承担。
- 哪些明确不由本模块承担。
- 本次可能影响哪些入口、组件、接口、数据、配置或测试。
- 需求中的模糊质量词或事件词如何落成可测定义，例如“及时”“稳定”“目录变更”“高频”“安全”等。
- 当前应使用轻量、标准还是增强输出。

这一步只用于确定范围和输出强度，不定稿 TOBE 决策。

### 3. 检查 ASIS 覆盖和阻塞

确认 ASIS 是否覆盖初步范围中的关键事实：

- 关键入口、流程、数据、配置、测试和外部交互是否有证据。
- 关键结论是否为事实，还是推断/待确认/阻塞相关。
- 规格漂移、低置信度或测试缺口是否影响 TOBE 定稿。

依赖 ASIS 阻塞项、低置信度结论或待确认关键事实的 TOBE 决策不得定稿。只能继续设计与该阻塞无依赖的独立部分。

如果待确认项会影响模块边界、核心方案、数据/权限/兼容策略、性能目标、验收阈值或任务拆分，必须立即向用户或上游负责人问询；若当前执行环境不能直接问询，则在成果物中标记为 `需前置确认`，并阻止依赖该问题的决策定稿。

### 4. 形成 TOBE 决策

围绕四条链输出最小充分设计：

- 边界与职责：本模块承担什么，不承担什么，是否需要更新上游设计或 `softWare.md`。
- 设计决策：新增、修改、复用、废弃、迁移或不做什么，为什么。
- 工程落点：改哪些文件、包、类、函数、API、数据、配置或开关。
- AICoding 任务：任务粒度、依赖、主要改动区域、验收标准、最小验证集。
- 可测定义：将影响验收的模糊质量词、事件触发条件和时效要求写成可测试的输入、阈值、观察点或断言。
- 流程/时序图：涉及多组件协作、跨模块交互、状态变化、写后读、异常路径或回滚时，至少输出一个 Mermaid 流程图或时序图，指导 AICoding 理解调用顺序与责任边界。

不要复制 ASIS 大段事实；只引用影响设计的 ASIS 结论编号和证据编号。

### 5. 展开被触发的风险专题

只展开被触发的风险专题；未触发时写明不适用原因。

- 架构防腐化：跨模块、跨层、依赖方向、外部模型、公共接口、共享状态变化。
- 安全增强：支付、隐私、凭证/密钥、权限边界、跨租户数据、外部输入、文件路径、命令执行、审计合规、AI/LLM 上下文污染。
- 性能增强：高频调用、批处理、大数据量、同步核心链路、启动路径、核心交易链路、明确 SLA、资源瓶颈、缓存/降级策略变化。
- 日志可观测性：主流程追踪、错误路径定位、日志密度、敏感信息脱敏、指标/告警。
- 兼容/迁移/回滚：接口签名、数据结构、配置默认值、迁移回填、灰度和回退。
- 并发/事务/幂等：状态变更、重复请求、重试、超时、补偿、数据一致性。

安全和性能采用两档强度：

- `普通分析`：默认强度，说明输入边界、权限/敏感信息、复杂度、I/O、内存、日志密度和验证方式；不适用项写明原因。
- `增强分析`：风险触发时启用。安全补攻击面、数据分级、信任边界、滥用场景、缓解措施和验证方式；性能补 ASIS 基线或现状估计、目标指标或 SLA、负载估算、瓶颈风险、验证方法和回退策略。

### 6. 更新目标成果物

加载 `references/tobe-output-template.md`，按输出模式更新目标成果物的 TOBE 章节。

- 轻量输出：合并章节，保留最小充分信息，空表用“不适用，原因...”替代。
- 标准输出：保留核心骨架，展开与本次需求相关的设计点和风险专题。
- 增强输出：补齐触发风险专题的专项分析和更严格验证设计。

返工时只更新受影响章节，并在 TOBE 变更记录中说明本轮变化。

## 阻塞规则

出现以下情况时标记 TOBE 阻塞或部分完成：

- ASIS 阻塞项直接影响本次 TOBE 决策。
- 需求/AR 与 ASIS 证据映射存在关键未覆盖项。
- 上游设计互相矛盾，导致无法判断本模块职责。
- 模块间交互设计缺失或冲突，导致接口、事件或数据归属无法确定。
- 关键兼容、迁移、权限、数据一致性、安全或性能策略无法从现有材料判断。

阻塞输出必须包含：阻塞原因、受影响的 TOBE 决策、需要回补的 ASIS 或上游输入、当前可继续设计的独立范围、不得定稿的设计决策、对 AICoding 的影响。

## 质量标准

结束前确认：

- 输出遵循共享方法论和当前输出模式。
- 证据链、边界链、执行链、风险链已闭合，或阻塞项已显性化。
- 每条重要 TOBE 决策引用需求/AR 来源和 ASIS 证据编号。
- 设计没有越过模块职责边界。
- AICoding 任务能落到具体文件或代码区域，并包含最小验证集。
- 触发的风险专题已展开；未触发的专题有不适用说明。
- 安全和性能按普通/增强两档选择了合适强度。
- 未决问题和阻塞项没有被伪装成设计结论。
