# 模块深度研究

## 执行原则

指定目标模块后，持续推进研究循环。`research-queue.md` 是唯一工作队列；检查点不是完成条件。

研究状态由 `research-state.md` 驱动。启动后状态为 `RUNNING`；只有触发停止条件并写入 `STOPPED` 与停止原因后，才允许结束当前运行。

若运行环境支持续跑调度，调度器必须在每次 Agent 返回后检查 `research-state.md`：状态仍为 `RUNNING` 时，将该返回视为阶段返回而非完成，并立即恢复同一任务继续研究。

本 skill 的职责是维持一个不断加深的研究循环：

```text
建队列 -> 选切片 -> 取证 -> 追踪 -> 反证 -> 沉淀 -> 更新覆盖 -> 扩展队列 -> 继续
```

资产结构见 `<skill-dir>/references/asset-structure.md`，研究策略见 `<skill-dir>/references/research-loop.md`。

只读代码、测试、文档、配置和 git 历史；只写模块认知资产；不得修改业务代码、提交代码或伪造无法验证的结论。

## 启动

1. 解析目标模块名称、目录、包路径或功能区域。
2. 定位或创建 `.sdd/project-understanding/modules/<module>/`。
3. 立即完成启动落盘：创建缺失资产，将 `research-state.md` 置为 `RUNNING`，并在 `research-queue.md` 写入至少一个可取证切片。
4. 读取 `research-state.md`、`research-queue.md`、`coverage-map.md`、`open-questions.md`、`study-log.md`、`method-notes.md` 和已有认知资产。
5. 若 `research-queue.md` 为空或只有粗粒度意图，基于代码入口、测试、配置、近期 git 历史、低置信结论、未解问题和高风险区域生成初始队列。
6. 从队列中选择价值最高且当前可取证的切片，立即开始研究。

启动落盘是前置条件。没有 `RUNNING` 状态和可恢复队列时，不得开始长时间阅读代码，也不得输出只包含意图的计划。

## 持续循环

每轮只研究一个切片，并把它推进到明确状态：`已完成`、`阻塞`、`延后` 或产生更深子切片。

每轮尽早、小额、增量落盘；不要把大量阅读和推理积压到运行末尾再一次性写入。

每轮执行：

1. 标记当前队列项为 `研究中`，在 `research-state.md` 记录当前切片。
2. 写明本轮要验证的假设、盲区或冲突。
3. 读取最相关的代码、测试、配置、文档和 git 历史。
4. 沿真实路径追踪入口、调用链、数据、状态、副作用和失败路径。
5. 用调用方、测试、提交历史、相邻实现或反向搜索挑战关键结论。
6. 按深度等级更新认知资产、证据索引、覆盖地图和开放问题。
7. 关闭或更新当前队列项，把新发现的分支、风险、低置信点、过期点和历史疑点放回队列。
8. 执行研究复盘，记录本轮研究方法暴露的问题，并修正后续队列和覆盖判断规则。
9. 写入 `study-log.md`，更新 `research-state.md`，继续选择下一项。

不得用一次总结替代持续研究。只要队列中仍有可取证切片，就继续下一轮。

运行状态为 `RUNNING` 时，不允许结束当前运行。若执行环境要求阶段性说明，只能把说明写入认知资产，然后继续工具调用和下一轮研究。

## 深度等级

切片研究至少标记当前达到的深度：

- `L0 Inventory`：识别文件、入口、测试、配置和依赖。
- `L1 Path`：追踪一条主运行路径，能说明入口到主要副作用。
- `L2 Behavior`：覆盖关键分支、异常路径、数据/状态变化和外部交互。
- `L3 Rule`：提取不变量、契约、风险、兼容要求和测试证据。
- `L4 History`：结合 git 历史识别设计原因、反复变更和变更规律。

核心路径、状态写入、跨模块契约、高风险区域不能停在 `L0` 或 `L1`；必须继续拆分子切片，直到达到 `L2+`，或记录为阻塞。

## 持续运行要求

每轮记录更新后必须继续推进：

1. 若 `research-queue.md` 中存在 `待研究` 或可恢复的 `延后` 切片，立即选择下一项。
2. 若队列为空，从 `coverage-map.md`、`open-questions.md`、`method-notes.md`、近期 git 历史和证据缺口生成新切片。
3. 若全部切片均为 `已完成`、`阻塞` 或不可取证的 `延后`，从覆盖不足、低置信、过期认知、测试缺口、历史变更、相邻调用方、配置入口、文档漂移和风险假设中拆出更小切片。
4. 如果某个方向需要修改代码或新增测试才能继续，把它写入 `open-questions.md` 或 `risks.md`，再切换到另一个只读可取证切片。
5. 只有满足停止条件时，才允许结束。

不得因为完成一轮研究、完成一个检查点、写完一组资产或产出阶段性总结而停止。

## 检查点记录

完成可交付检查点时，只记录到认知资产，不把检查点当成运行结束。检查点必须至少满足一项：

- 一条核心运行路径达到 `L2+`。
- 一个关键数据/状态模型达到 `L2+`。
- 一个不变量、契约、风险或变更规律达到 `L3+`。
- 一个历史原因或反复变更模式达到 `L4`。
- 一个过期认知被修正并补齐证据索引。
- 一个阻塞被明确记录，且队列中已安排可继续的非阻塞切片。

检查点记录后立即回到队列。除非触发停止条件，不得把检查点当作任务完成。

## 方法复盘与修正

每轮研究结束前必须复盘研究方法本身：

- 本轮是否停在了表层文件摘要，而没有形成行为理解。
- 本轮证据是否足以支撑结论状态。
- 本轮是否遗漏了调用方、测试、配置、历史或反例路径。
- 本轮资产组织是否让下一轮更容易继续。
- 本轮队列是否产生了更具体、更可取证的下一步。

如果发现研究方法存在缺口，不通过列举特例修补；应把缺口抽象成新的队列生成规则、深度升级要求、覆盖判断规则或资产写入约束，记录到 `method-notes.md`，并反映到后续队列和覆盖地图中。

方法复盘与修正只更新当前模块认知资产中的运行策略和队列，不修改 skill 文件本身。

## 停止前记录

只有以下情况可以停止：

- 收到中断指令。
- 环境或工具限制导致无法继续读取必要材料。
- 上下文即将耗尽，必须先记录当前研究状态。
- 目标模块无法定位。
- 当前研究必须依赖外部系统、生产数据、团队裁决或代码修改才能继续，且仓库内不存在任何可转向的只读切片。

以下情况不构成停止条件：

- 当前切片完成。
- 当前队列暂时为空。
- 下一步最好新增测试或修改代码。
- 已发现的问题已经足够明确。
- 已形成一个阶段性总结。

存在 `研究中`、`待研究`、可恢复的 `延后` 队列项，或能从现有资产生成新的只读切片时，不满足停止条件。

停止前必须更新：

- `research-state.md`：运行状态置为 `STOPPED`，写明停止原因。
- `research-state.md`：当前切片、已完成动作、待记录发现、恢复入口。
- `research-queue.md`：所有队列项状态和下一步动作。
- `coverage-map.md`：最新覆盖状态。
- `study-log.md`：本轮研究记录。
- `method-notes.md`：本轮形成的方法反馈。
- `open-questions.md`：仍需外部输入的问题。

## 续跑规则

本 skill 的持续性由两层共同保证：

- 内层 Agent：按队列持续研究，并维护 `research-state.md`。
- 外层调度器：在 Agent 每次返回后读取 `research-state.md`，决定是否恢复运行。

调度规则：

1. `RUNNING`：返回内容只视为阶段返回或检查点，立即恢复 Agent，继续当前模块研究。
2. `STOPPED`：读取停止原因；若停止原因符合本 skill 的停止条件，才视为完成。
3. 状态缺失或不合法：恢复 Agent，要求先修复 `research-state.md`，再继续研究。

不要用一次 Agent 返回判断研究完成；只用 `research-state.md` 的状态和停止原因判断。

## 质量要求

- 重要结论必须能反查证据。
- 必须区分 `FACT`、`INFERRED`、`CONFIRMED`、`STALE`、`BLOCKED`。
- 每轮研究必须关闭当前队列项或产生更具体的子队列项。
- 只要 `research-state.md` 为 `RUNNING`，不得结束当前运行。
- 发现资产与代码冲突时，先标记 `STALE`，再写入修正结论和冲突证据。
- 每轮研究必须留下对下一轮有用的方法反馈，而不是只留下内容总结。
- 研究产物必须能让下一次执行从 `research-state.md` 和 `research-queue.md` 继续。
