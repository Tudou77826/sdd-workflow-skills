# 深度研究循环细则

本文件补充 `SKILL.md` 的运行流程细则。执行时先按 `SKILL.md` 的运行流程推进，再按本文件补充队列、记录和增量处理细节。

深度研究由 `research-queue.md` 驱动。队列承载下一步候选切片；其他文件提供证据、覆盖状态和阻塞信息。

## 队列生成来源

队列为空或剩余切片过粗时，从以下来源生成候选切片：

- 模块入口：API、Controller、Handler、Job、Consumer、CLI、公开函数。
- 测试入口：单元测试、集成测试、Fixture、Mock、快照、契约测试。
- 配置入口：环境变量、Feature Flag、路由、依赖注入、构建配置。
- 近期历史：最近变更、反复修复、回滚、重命名和迁移。
- 增量变化：`baseline_commit..HEAD` 中命中证据锚点、入口、测试、配置或关键路径的变化。
- 认知缺口：`coverage-map.md` 中低深度、低置信、过期和阻塞区域。
- 开放问题：`open-questions.md` 中仍可通过仓库取证的问题。
- 方法反馈：`method-notes.md` 中记录的薄弱探索模式。

## 增量刷新细则

运行流程中的增量刷新负责把代码变化转化为复核队列，不把 diff 直接写成最终结论。

刷新细则：

1. 读取 `incremental-state.md` 中的 `baseline_commit`；缺失时，记录当前 `HEAD`、补建 `incremental-state.md`，并生成“补齐关键结论证据锚点”的队列项。
2. 若 `baseline_commit` 等于当前 `HEAD`，跳过 diff 吸收，继续常规队列。
3. 若二者不同，读取 `git log --oneline baseline_commit..HEAD`、`git diff --name-status baseline_commit..HEAD`，并针对相关文件读取必要 diff。
4. 写入简短变更画像：变化范围、可能意图、涉及入口、测试、配置、数据结构、调用链和风险区域。
5. 用变更画像反查 `evidence-index.md`、`coverage-map.md` 和各认知资产中的证据锚点，找出可能过期的结论。
6. 对命中的结论先标记 `STALE` 或生成复核队列项，再读取当前代码完整上下文、调用方、被调用方、测试和配置。
7. 复核后更新结论状态、证据锚点、`last_verified_commit`、覆盖地图和研究日志。
8. 只有命中的复核队列已处理、阻塞或转入开放问题后，才更新 `incremental-state.md` 的 `baseline_commit`。
9. 进入队列生成或选题步骤。

若 diff 没有命中任何既有结论，不视为无影响；应记录为证据索引缺口，并生成补充证据锚点或覆盖地图的队列项。

## 选题排序

选择切片时按以下顺序比较：

1. 能否立即从仓库取证。
2. 是否命中已有重要结论的证据锚点。
3. 是否位于核心运行路径。
4. 是否影响数据、状态、事务、幂等或外部契约。
5. 是否存在 `INFERRED`、`STALE` 或 `BLOCKED` 认知。
6. 是否近期频繁变更、修复或回滚。
7. 是否测试覆盖少但影响面大。

## 深度升级

研究切片时按深度升级，不要停在文件罗列：

- `L0 Inventory`：识别文件、入口、测试、配置和依赖。
- `L1 Path`：追踪一条主运行路径，说明入口到主要副作用。
- `L2 Behavior`：覆盖关键分支、异常路径、数据/状态变化和外部交互。
- `L3 Rule`：提取不变量、契约、风险、兼容要求和测试证据。
- `L4 History`：结合 git 历史识别设计原因、反复变更和变更规律。

核心路径、状态写入、跨模块契约、高风险区域最低目标是 `L2`。发现不变量、兼容逻辑或反复变更时，升级到 `L3` 或 `L4`。

## 切片研究闭环

切片研究内部必须形成闭环：

```text
队列选题 -> 假设/盲区 -> 证据读取 -> 路径追踪 -> 反证检查 -> 结论更新 -> 覆盖更新 -> 队列更新
```

如果证据不足，不要扩大结论；把问题写入 `open-questions.md`，并在 `research-queue.md` 中安排下一步取证。

## 启动落盘内容

启动落盘写入一组最小可恢复记录，再开始长时间阅读：

- 创建缺失的认知资产骨架。
- `research-state.md` 写入 `RUNNING`、当前动作和恢复入口。
- `incremental-state.md` 写入当前基线、待复核变化和恢复入口；没有历史基线时，先把补齐证据锚点放入队列。
- `research-queue.md` 写入至少一个可取证切片；不知道入口时，先写入“定位入口”的切片。
- `coverage-map.md` 记录尚未覆盖的主要区域。

启动落盘不要求已有结论；它保证任务被中断后可以续跑。

## 记录更新内容

每轮记录至少更新：

- 采用小额增量写入，优先记录当前状态、证据锚点和下一步，不等待完整结论一次性成文。
- 当前队列项状态更新为 `已完成`、`阻塞`、`延后`，或拆出更具体的子切片。
- `study-log.md` 记录本轮切片、证据、反证、结论和资产更新。
- 受影响的认知资产写入结论和证据锚点。
- 增量复核命中的结论写入 `last_verified_commit`；未完成复核的命中项保留 `STALE`。
- `coverage-map.md` 更新区域深度、置信状态和盲区。
- `incremental-state.md` 更新本轮基线、变更画像、待复核项和恢复入口。
- `research-state.md` 更新当前游标和恢复入口。
- 新发现的问题进入 `open-questions.md` 或 `research-queue.md`。
- `method-notes.md` 记录本轮暴露的研究方法问题和下一轮应采用的修正策略。

## 方法复盘

每轮记录更新前，检查本轮是否出现以下问题：

- 只总结文件职责，没有追踪真实行为。
- 只确认主路径，没有寻找失败路径、边界分支或反例。
- 只引用代码，没有交叉检查测试、配置、调用方或历史。
- 生成的队列项过大，下一轮无法直接取证。
- 覆盖地图只记录“看过”，没有记录深度和盲区。

出现问题时，不为单个现象写硬编码规则；把它提升为后续研究策略，写入 `method-notes.md`，并用它重排或拆分 `research-queue.md`。

## 队列字段

`research-queue.md` 中的候选切片至少记录：

- `ID`：稳定编号。
- `切片`：具体到可取证的路径、场景、组件或问题。
- `价值`：高 / 中 / 低。
- `状态`：待研究 / 研究中 / 已完成 / 阻塞 / 延后。
- `深度`：L0 / L1 / L2 / L3 / L4。
- `触发原因`：核心路径、低置信、近期变更、风险、过期认知、测试缺口或未解问题。
- `证据入口`：代码、测试、配置、文档或提交。
- `下一步`：下一次打开该切片时要做的具体动作。

## 续推来源

未满足停止条件时，从以下来源继续生成或选择切片。当前方向需要代码修改或新增测试时，记录为问题或风险，并转向另一个只读可取证切片。

`research-state.md` 为 `RUNNING` 时，不能用阶段性说明代替下一轮研究。只有 `research-state.md` 已写入 `STOPPED` 和停止原因，才允许结束当前运行。

常见来源：

- 覆盖地图中 `L0/L1` 或低置信区域。
- 已完成切片中暴露出的分支、异常路径、调用方和测试缺口。
- `open-questions.md` 中可以通过仓库继续取证的问题。
- `method-notes.md` 中记录的薄弱探索模式。
- 近期 git 历史中反复修改或回滚的区域。
- 相邻调用方、配置入口、文档描述和测试缺口暴露出的只读研究切片。

完成检查点只意味着可以记录成果，不意味着可以结束研究。

## 检查点记录

检查点应记录：

- 本轮研究切片和达到深度。
- 新增或修正的关键认知。
- 关键证据位置。
- 已更新的认知资产。
- 新增队列项或关闭的队列项。
- 仍然不确定的问题。

记录后继续下一轮。只有停止条件成立时才结束当前运行。
